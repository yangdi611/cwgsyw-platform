# v0.19.0 — 2026年6月28日

本次更新围绕**硬件/机房/数据库**配置管理展开，新增 12 类内置模型、2D 机柜视图、维保状态自动计算、批量编辑等能力。

## 新功能

### 硬件/数据库模型组（12 个内置模型）

在「模型管理」页可见：

- **硬件组** — storage（存储）、san_switch（SAN 交换机）、net_switch（网络交换机）、rack（机柜）
- **数据库/中间件组** — mysql、ob（OceanBase）、oracle、redis、kafka、nginx
- **机房组** — building（楼宇）、idc_room（机房）

每类模型预置 **50 个属性组 + 292 个标量字段**，含基础信息（序列号、品牌型号、采购日期、管理 IP）、硬件规格（CPU/内存/磁盘/端口数/槽位数）、维保信息、扩展属性等。所有模型均已配置 **bk_mainline 包含层级关系**（building ⊃ idc_room ⊃ rack ⊃ host/storage/switch）与 **run_on 运行关系**（app/mysql/redis 运行在 host 上），在拓扑图中自动呈现上下游依赖。

### 机柜 2D 视图

机柜（rack）实例详情页新增「机柜视图」标签（仅 rack 模型可见）：

- **竖向刻度** — U 位从下往上（1U 底、42U 顶），左侧显示 U 刻度
- **设备块** — 每台设备按 `u_start/u_end` 定位，高度 = 跨 U 数；底色取模型配色（CI 模型管理页设置的 color 字段）；块上显示设备名 + 资产编号
- **点击跳转** — 设备块可点击，直达该设备详情
- **添加设备入口** — 视图右上角「添加设备」按钮，跳转关联新建流程（选 host/storage/switch → 自动建 `rack_contains_*` 边 → 可在边 metadata 或设备 attrs 内填 U 位信息）
- **布局告警** — U 位越界（超出机柜 rack_height_u）、区间重叠、未登记 U 位（uStart/uEnd 为空）、invalid_range（u_end < u_start）时，顶部红色告警区飘出提示
- **未定位列表** — 已关联但未填 U 位的设备，集中列在视图右侧「未定位设备」区，避免遗漏

后端：`GET /api/cmdb/rack/{id}/layout` 接口（`RackLayoutService`），聚合机柜及其 `rack_contains_*` 边直连的设备，算 U 占用并产生告警。

### 维保状态自动计算

所有模型（含 host 等原有模型）均已追加「维保信息」属性组（9 个字段：维保厂商、维保级别、维保状态、起始/到期日期、合同号、联系人、响应SLA、备注），其中 **维保状态（maint_status）** 现改为派生只读字段：

- 后端在 CI 实例详情接口（`CiInstanceQueryService.getDetail`）内，读取实例的 `maint_expire`（维保到期日期），与今天比较：
  - 已到期 → `expired`（已过保）
  - 30 天内到期 → `expiring`（即将到期）
  - 30 天后到期 → `active`（在保）
- 未填 `maint_expire` 或无法解析时不注入派生字段，前端无徽章
- 派生字段以 `_maint_status_derived` 注入 `fieldsData`（`_` 前缀，与用户 field_key 正则 `^[a-z][a-z0-9_]*$` 天然隔离，用户无法在模型管理页创建 `_` 开头的属性）

前端：
- **详情页徽章** — 实例详情页「基本信息」标签页顶部，红黄绿徽章：绿「维保在保 · 到期 2027-05-30」/ 黄「即将到期 · 到期 2026-07-15」/ 红「已过保 · 到期 2026-01-10」。无 `maint_expire` 不显示
- **列表标色**（未来迭代）— 实例列表页的「维保状态」列可按派生值标色
- **维保到期看板**（未来迭代）— 单独看板汇总全平台即将到期/已过保设备，按模型分组，定期巡检用

### 批量编辑（实例列表页）

实例列表页（`/cmdb/instances/by-model/{modelCode}`）新增多选 + 批量编辑：

- **多选复选框** — 表格左侧出现复选框列（需 `cmdb_instance:update` 权限），勾选一条或多条
- **批量编辑按钮** — 勾选后右上角浮出「批量编辑（N）」按钮
- **弹窗覆盖字段** — 点击打开批量编辑弹窗，选择一个公共标量字段（owner/env/criticality/维保相关/云资源专属等）+ 输入新值 → 「应用」
- **逐条独立审计** — 后端 `POST /api/cmdb/instances/batch-update` 逐条调用 `CiInstanceCommandService.update`，每条产生独立审计日志与变更记录；单条失败（唯一冲突/校验失败）不阻断其余，结果页汇总成功/失败数 + 失败明细
- **仅标量字段** — table 类型字段（结构化多值）不在批量编辑范围

### 云资源池（resource_pool）运行时属性补齐

K8s 聚合模型 `resource_pool`（用户在模型管理 UI 运行时创建，不是 Flyway 种子，实库查证 `is_built_in=false`）现由 **启动期 seeder**（`RuntimeModelBaselineSeeder`，`@Order(101)`，紧跟 WikiManualSeeder）自动补齐与其他预制模型对齐的属性：

- 「维保信息」组 + 维保 9 字段（同上）
- 「容量与配置」组内追加 6 个云池专属标量：k8s_version、api_server、cni_plugin、runtime（enum：containerd/docker/cri-o）、ha_enabled、monitor_enabled

语义：**只增不改不删**。每个属性/属性组 insert 前用 `selectCount` 判存在，已存在则跳过。不存在的模型直接跳过（绝不创建模型——创建是用户的职责）。幂等，多次启动无副作用。

⚠️ **MyBatis 自动填充陷阱**：`BaseEntity.createdBy/updatedBy` 是 `@TableField(fill=INSERT)`，靠 `MetaObjectHandler` 从请求上下文取当前用户。**ApplicationRunner 无请求上下文 → 填 null → 违反 NOT NULL**。解法：seeder 里手动 `setCreatedBy(0L)/setUpdatedBy(0L)`（仿 WikiManualSeeder 的 `SYSTEM_USER=0L`）。

## 改进

- **关联定义覆盖全域** — 新增 12 个关联 def（`rack_contains_host/storage/san_switch/net_switch`，`host_runs_mysql/ob/oracle/redis/kafka/nginx`），支持包含层级与运行关系拓扑图自动推导

## 技术细节

- **V56 迁移**（`V56__cmdb_seed_models_baseline.sql`，484 行）：12 模型 + 2 分组 + 50 属性组 + 292 属性 + 12 关联 def，全 `ON CONFLICT DO NOTHING`，幂等可重跑。rack 模型 `enable_2d_view=TRUE`。Flyway 实跑 55→56 成功
- **RuntimeModelBaselineSeeder**：仅针对 `resource_pool` 等运行时模型（`TARGET_MODELS` 列表），避免 Flyway 无法假定其存在的困境
- **RackLayoutService + findRackMembers**：`CiInstanceRelMapper.findRackMembers` 查 `rack_contains_*` 边直连设备，`NULLIF(attrs->>'u_start','')::int` 防空串转换异常；Service 端构造 occupancy 数组检测重叠
- **机柜视图前端**（`RackElevationView.tsx`）：单 U 22px，机柜块 absolute 定位，U 行从上往下渲染（顶部 = rackHeightU U，底部 = 1U）
- **批量编辑**（`BatchUpdateInstanceRequest` + `BatchUpdateResultVO`）：逐条 try-catch，失败收集 `FailItem{id, error}`，不抛异常
- **维保派生注入**（`injectMaintStatusDerived`）：仿 `injectResourcePoolDerivedFields` 模式，`_` 前缀只读键，详情/列表复用
