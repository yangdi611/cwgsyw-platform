# v0.20.0 — 2026年6月28日

本次更新完成 CMDB 硬件配置管理的 **P2/P3 能力补全**（table 结构化字段、JSON 批量导入、克隆、端点连接），并完成一项 **全局 API 命名规范统一**（snake_case → camelCase）的架构治理。

## 新功能

### table 类型字段（结构化多值）

CMDB 属性新增 **table（表格）类型**，用于一个实例下的多行结构化数据（如主机的多块网卡、多个文件系统、交换机的多个端口）：

- **子列 schema** — 每个 table 字段定义若干子列，每列有 key/name/type（singlechar/int/enum/date 等）/required/options。模型管理页提供「插入模板」快速生成常见结构（网卡 nics、文件系统 filesystems、端口 ports 等）
- **行级编辑** — 实例详情页对 table 字段以表格形式增删行、逐格编辑；每行自动带稳定 `row_id`（后端补生成，作为端点连接与基线对齐的锚点）
- **后端校验** — `CiFieldSchemaValidator` 对每行逐列校验：必填子列缺失、enum 越界、int 非法值均报错；`row_id` 为系统列不强制用户填；兼容旧「裸数组」schema
- **导入兼容** — JSON/NDJSON 导入可直接携带 table 数组（见下）

### JSON / NDJSON 批量导入

实例列表页「导入」对话框新增 **格式切换**，除 CSV 外支持 JSON 数组与 NDJSON（一行一对象）：

- **走标准命令链路** — 不复用 CSV 的行解析；解析后构造标准请求调 `CiInstanceCommandService.create/update`，自动获得 table 校验 + 唯一性 + 审计 + 变更记录
- **结构化字段** — 顶层 `name` 为实例名，其余键（含 table 数组，每行带 `row_id`）进入字段数据。可表达 CSV 无法表达的多值嵌套
- **三档导入模式** —
  - `merge`（合并，默认）：只更新本次出现的顶层键，table 整键覆盖
  - `replace_fields`（字段替换）：出现即覆盖
  - `baseline_replace`（基线重导）：table 字段按 `row_id` 全量对齐——缺失行删除、新行插入、同 row_id 更新。用于每月采集脚本无脑重导反映拔卡/卸盘/下线
- **唯一键匹配** — 可指定唯一键字段（如 `asset_no`）判 create/update，留空按 name 匹配
- **逐条容错** — 单对象失败（唯一冲突/校验失败）不阻断其余，结果页汇总成功/失败明细

后端：`JsonImportService` + `POST /api/cmdb/instances/import/json/preview|execute`。

### 实例克隆

实例列表抽屉新增「克隆」：

- 复制源实例全部字段，清空唯一字段与 table 行的 `row_id`（避免冲突）
- 自动以 `-副本` 后缀去重命名
- `POST /api/cmdb/instances/{id}/clone`

### 端点连接（ci_endpoint_link，端口级连接）

新增 **端口级连接**能力，描述设备间的物理/逻辑链路（网口↔交换机口、HBA↔SAN 口、存储 LUN↔主机）：

- **唯一事实源** — `ci_endpoint_link` 表为连接的唯一写入入口；增删连接自动维护 `connect` 类 **managed 镜像边**（拓扑图可见），并对镜像边做引用计数（link_count）
- **端点锚定** — 连接两端锚定到本实例 table 字段（nics/ports/luns 等）的某一行 `row_id`，精确到端口
- **保护规则** — managed/connect 类边不允许手动增删改（只能通过连接接口维护），防止镜像边与事实源脱钩
- **设备详情卡片** — 拥有 table 字段的设备详情页出现「端口连接」卡片，可视化新建/解除连接
- **4 个内置 connect 定义** — host↔net_switch、host↔san_switch、san_switch↔storage、storage↔host(LUN)

后端：`EndpointLinkService` + `POST/DELETE /api/cmdb/endpoint-links`，`GET /api/cmdb/endpoint-links/by-instance/{id}`。

### 设备「所在机柜」反向装机

设备（host/storage/switch 等）详情页新增「所在机柜」卡片：

- 选择机柜建立 `rack_contains_*` 反向关联（rack→device）
- 同时把起始/结束 U 位写入**设备自身属性**（`attrs.u_start/u_end`，与机柜视图读取源一致），填了即在机柜视图正确定位
- `GET/POST /api/cmdb/instances/{id}/relations/reverse-defs|reverse`

## 改进

### 全局 API 命名统一为 camelCase（架构治理）

历史上后端全局配置 `SNAKE_CASE`，CMDB 模块又用 `@JsonNaming` 局部覆盖回 camelCase，且改造只做了一半，导致接口字段命名 camel/snake/类内混合三层并存，前端读取易出错（本次机柜视图 `modelId`/`uStart` 读空即由此引发）。

本次彻底统一：

- **移除**后端全局 `property-naming-strategy: SNAKE_CASE`
- **删除**全部 23 处 `@JsonNaming` 注解
- **前端 58 个文件**的 TS interface、字段读取、POST/PUT body 统一改 camelCase
- **过渡兼容** — 非 cmdb 旧请求 DTO 加 54 处 `@JsonAlias`，camelCase 与 snake_case 双接受，旧调用不会断
- **结果** — Java `modelId` → JSON `modelId` → 前端 `modelId` 三层同名，零转换

> 例外：CMDB 实例 `fieldsData`（`attrs` JSONB）**内部的键**是用户定义的属性 field_key（如 `inner_ip`/`cpu_cores`），属数据内容不是 API 字段，保持原样。

### 机柜视图字段读取修复

修复机柜视图设备链接变 `by-model/undefined/21`、已填 U 位仍报「未定位」的问题（根因为上述命名不一致 + `uStart`/`uEnd` 这类单字母前缀字段的 Jackson getter 命名歧义，已用 getter 上 `@JsonProperty` 钉死）。

## 技术细节

- **V58 迁移**（`V58__ci_endpoint_link.sql`）：`ci_endpoint_link` 表 + 源/目的端两个部分唯一索引（防同一端点重复连接）+ 两个查询索引 + 4 个 connect 关联 def，全 `IF NOT EXISTS`/`ON CONFLICT` 幂等。Flyway 实跑 56→58 成功
- **EndpointLinkService**：连接增删在事务内同步 upsert/递减 connect 镜像边的 link_count，count 归零才软删镜像边
- **JsonImportService**：NDJSON 逐行 / JSON 数组自动识别；`baseline_replace` 读旧实例 fieldsData 按 row_id 三路对齐后整键提交
- **CiFieldSchemaValidator**：table 行类型 + 子列必填/enum/int 校验；8 个单元测试覆盖
- **命名统一**：移除 `application.yml` 全局策略 + 删 @JsonNaming；`RackDeviceVO` 的 `uStart`/`uEnd` getter 保留 `@JsonProperty` 解决 Lombok `getUStart()` 被推断为 `ustart` 的歧义
- **测试**：新增 14 个后端单元测试（table 校验 8 + JSON 导入解析 6），后端编译 + 前端 `next build` 全过
