三个关键技术选型

这三个决策影响最大，请逐一确认

决策 1 — 前端框架
A

Next.js 15 + React 19 推荐

App Router、SSR/SSG 混合渲染、Server Actions。2024-2025 最主流，生态最完整。shadcn/ui + Tailwind v4 组件库，开箱即用企业级 UI。

适合：需要 SEO 或未来对外开放的场景；部署需要 Node.js 服务
B

Vite + React 19 (SPA)

纯客户端渲染，构建产物是静态文件，Nginx 直接托管。配合 TanStack Router + TanStack Query，架构更简单，运维更轻量。

适合：内部工具，不需要 SSR；部署最简单，一个 Nginx 容器搞定
决策 2 — 审批流引擎
A

Flowable 7 推荐

Java 生态最成熟的流程引擎，Spring Boot 无缝集成，支持 BPMN 2.0 标准。支持或签/会签/条件分支，可视化流程配置。社区活跃，文档完整。

代价：学习成本略高，数据库表多（约30张）
B

自研轻量审批流

自己实现状态机：节点表 + 实例表 + 流转记录表。灵活可控，表结构简单，改起来方便。但需要自己实现或签/会签/条件分支逻辑。

代价：前期开发量大，随业务复杂度增长维护成本高
决策 3 — 文档生成方案
A

Apache POI (Word) + iText/OpenPDF (PDF) 推荐

Word 模板用书签占位符填充，导出 .docx 格式完全保留原始排版。PDF 由 Word 转换或独立生成，水印通过 iText 叠加。Java 生态最成熟方案。

在线编辑器存 HTML，导出时映射到 Word 模板
B

LibreOffice 容器转换

后端起一个 LibreOffice 无头容器，Word 直接转 PDF，排版还原度最高。但需要额外维护一个容器，内存占用较大（约 512MB）。

适合：对 PDF 排版还原度要求极高的场景
