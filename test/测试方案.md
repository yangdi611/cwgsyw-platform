# cwgsyw-platform 测试方案 (第一部分：总览与环境)

## 1. 测试目标与范围

### 覆盖模块

本次手工测试覆盖平台全部 10 个功能模块：

| 编号 | 模块 | 说明 |
|------|------|------|
| 1 | 认证/RBAC | 登录/登出、JWT 令牌、权限拦截、角色分级 |
| 2 | 用户/组管理 | 用户 CRUD、组管理、角色分配 |
| 3 | 日报系统 | 日报提交/编辑/审批/导出、Flowable 工作流 |
| 4 | 设备密码库 | 设备凭据 CRUD、AES-256-GCM 加解密、密码查看审计 |
| 5 | 通知中心 | 站内信收/发、邮件发送、通知铃铛轮询 |
| 6 | AI 网关 | 多供应商配置、AI 调用日志、对话 |
| 7 | 变更文档 | 双模板动态表单、审批流、Word/PDF 导出、自动归档 |
| 8 | CMDB | 模型/属性/关联元数据、CI 实例 CRUD、实例关联、拓扑视图 |
| 9 | 审计日志 | 操作日志查询、时间范围筛选、操作人/类型过滤 |
| 10 | 共享文件库 | 文件夹树、文件上传/下载/预览、组可见性、pandoc 转换 |

### 测试维度

每个模块从以下四个维度进行覆盖：

| 维度 | 说明 |
|------|------|
| 正常流程 | 功能的主要成功路径，如登录→操作→退出、创建→查询→编辑→删除 |
| 异常边界 | 空值、超长输入、非法格式、重复提交、并发操作、文件类型/大小限制 |
| 权限隔离 | 不同角色（super_admin / admin / group_leader / member / viewer）对同一功能的访问差异 |
| UI 交互 | 页面渲染正确性、按钮状态（禁用/隐藏）、加载态、空态、错误提示、响应式布局 |

### 测试方法说明

本文档为**纯手工测试方案**，不涉及自动化测试执行代码。所有测试用例由测试人员按照文档步骤手动执行，结果人工记录。自动化测试（Playwright MCP / Bruno API）归入 `test/e2e/` 和 `test/api/` 目录另行管理。

## 2. 测试环境

### 架构概览

采用 Docker Compose 本地六容器架构，所有服务在同一台机器上通过容器互联。

### 容器清单

| 容器名 | 镜像 | 端口 | 用途 |
|--------|------|------|------|
| postgres | postgres:16-alpine | 5432 | 关系数据库，存储所有业务数据 |
| redis | redis:7-alpine | 6379 | 缓存、Session 存储、消息队列 |
| minio | minio/minio:latest | 9000 (API) / 9001 (Console) | 对象存储，存储上传文件、导出文档 |
| backend | custom (Dockerfile) | 8080 | Spring Boot 3.4.5 API 服务 |
| frontend | custom (Dockerfile) | 3000 | Next.js 15 前端应用 |
| nginx | nginx:alpine | 80 | 反向代理，统一入口 |

### 启动方式

```bash
# 在项目根目录执行
docker compose up -d

# 查看各容器状态
docker compose ps

# 查看后端日志（调试用）
docker compose logs backend -f
```

### 访问地址

| 入口 | URL |
|------|-----|
| 前端界面 | `http://localhost:3000` |
| 后端 API 基础路径 | `http://localhost:8080/api` |
| MinIO Console | `http://localhost:9001` |
| PostgreSQL 直连 | `jdbc:postgresql://localhost:5432/cwgsyw_platform` |

### 数据库初始化

Flyway 迁移在 backend 容器启动时自动执行，无需手动干预。当前已完成 V1-V17 共 17 个迁移版本，覆盖所有模块的表结构创建和种子数据。

## 3. 测试账号矩阵

### 角色权限对照表

| 角色代码 | 角色名 | 用户名 | 密码 | groupScope | 关键权限 |
|----------|--------|--------|------|------------|----------|
| super_admin | 超级管理员 | superadmin | Admin@123 | platform | 全平台所有权限，可管理所有租户/组/用户，无任何限制 |
| admin | 管理员 | admin | Admin@123 | tenant | 租户级全部管理权限，可管理本租户内所有资源和用户 |
| group_leader | 组长 | leader1 | Leader@123 | group | 本组管理权限 + 审批权限（日报审批、变更审批），只能查看和操作本组成员数据 |
| member | 组员 | member1 | Member@123 | group | 基本读写权限，可创建/编辑自己的数据，只能查看本组可见内容 |
| viewer | 只读 | viewer1 | Viewer@123 | group | 只读访问权限，无任何写操作入口，页面按钮处于禁用/隐藏状态 |

### 账号说明

- **superadmin** 由 V3 迁移脚本通过 SQL 直接插入种子数据，Docker Compose 启动后立即可用。
- **admin / leader1 / member1 / viewer1** 四个账号需在测试前由 superadmin 登录系统后手动创建并分配相应角色。
- 以上密码均为开发/测试环境专用，生产环境必须替换。

### 创建测试账号步骤

1. 以 `superadmin / Admin@123` 登录系统
2. 进入 **系统管理 → 用户管理**，依次创建 admin、leader1、member1、viewer1
3. 在 **系统管理 → 角色管理** 中为每个用户分配对应角色
4. 对于 group_leader / member / viewer，需先创建测试组，再将用户加入该组
5. 验证每个账号能正确登录，且 sidebar 菜单项、操作按钮与角色权限一致
