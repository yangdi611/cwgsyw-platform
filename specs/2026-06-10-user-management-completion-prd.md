# PRD: 用户管理功能补全（phone/groups/搜索/审计日志）

## 背景

用户管理模块已实现基础 CRUD 功能，但存在以下功能缺口：
1. **手机号字段未暴露**：User 实体有 phone 字段，但创建/编辑接口和前端表单均未使用
2. **组织关联缺失**：用户可关联 group（通过 groupId），但用户创建/编辑界面无法选择所属组织
3. **搜索功能未实现**：前端有搜索输入框，但 keyword 参数未传递给后端，后端也未实现搜索逻辑
4. **审计日志缺失**：用户增删改操作未记录审计日志，无法追溯变更历史

本次需求目标是补齐这些功能，使户管理模块达到生产可用的完整性。

## 用户故事

- 作为**系统管理员**，我想在创建用户时填写手机号，以便在需要时通过短信或电话联系用户
- 作为**系统管理员**，我想在创建/编辑用户时指定所属组织，以便按组织架构管理权限和数据隔离
- 作为**系统管理员**，我想通过用户名、真实姓名或邮箱搜索用户，以便快速定位目标用户
- 作为**审计员**，我想查看用户的创建、修改、删除操作记录，以便追溯变更历史和满足合规要求
- 作为**系统管理员**，我想在审计日志中看到操作前后的数据对比，以便了解具体变更内容

## 验收标准

### 1. 手机号字段补全

- [ ] CreateUserRequest DTO 新增 `phone` 字段（可选，正则校验格式）
- [ ] UpdateUserRequest DTO 新增 `phone` 字段（可选）
- [ ] UserDetailVO 返回 `phone` 字段
- [ ] UserService.create() 方法保存 phone 到 User 实体
- [ ] UserService.update() 方法支持更新 phone 字段
- [ ] 前端 UserDialog 组件新增手机号输入框（位于邮箱下方）
- [ ] 前端用户列表页（/users）表格新增"手机号"列
- [ ] 手机号格式校验：支持中国大陆 11 位手机号（1 开头）或留空

### 2. 组织关联功能

- [ ] CreateUserRequest DTO 新增 `groupId` 字段（可选，Long 类型）
- [ ] UpdateUserRequest DTO 已包含 groupId（确认现有实现）
- [ ] UserDetailVO 返回 `groupId` 和 `groupName` 字段
- [ ] UserService.getDetail() 方法查询并返回关联的组织名称
- [ ] 前端 UserDialog 组件新增"所属组织"下拉选择器
- [ ] 下拉选择器加载当前租户的所有组织列表（调用 GET /api/groups）
- [ ] 选择器支持清空（groupId = null，表示未分配组织）
- [ ] 前端用户列表页表格新增"所属组织"列，显示组织名称
- [ ] 用户创建时指定组织后，用户的 groupId 正确保存
- [ ] 用户编辑时修改组织，groupId 正确更新

### 3. 用户搜索功能

- [ ] 后端 UserController.list() 接口接收 `keyword` 参数（可选，String）
- [ ] 后端 UserService.list() 方法支持按 keyword 模糊搜索
- [ ] 搜索范围：username、real_name、email 三个字段（OR 逻辑）
- [ ] 前端用户列表页将 keyword 状态传递给 API 请求
- [ ] 搜索输入框有防抖处理（300ms），避免频繁请求
- [ ] 清空搜索框时自动重置到第 1 页
- [ ] 搜索结果为空时显示"未找到匹配的用户"提示

### 4. 审计日志记录

- [ ] 用户创建操作记录审计日志（module="user", action="create"）
- [ ] 用户更新操作记录审计日志（module="user", action="update"）
- [ ] 用户删除操作记录审计日志（module="user", action="delete"）
- [ ] 审计日志包含操作前数据（beforeJson）：创建时为 null，更新/删除时为变更前快照
- [ ] 审计日志包含操作后数据（afterJson）：创建时为完整用户信息，更新时为变更后快照，删除时为 null
- [ ] 审计日志记录操作人 ID（operatorId）和操作时间（createdAt）
- [ ] 敏感字段（password）在审计日志中脱敏显示为 "***"
- [ ] 前端审计日志页（/admin/audit）可按 module="user" 筛选查看用户操作记录

## 交互流

### 创建用户（完整版）

1. 用户进入 `/users` 页面，点击"+ 新建用户"按钮
2. 系统弹出"新建用户"对话框
3. 用户填写表单：
   - 用户名（必填）
   - 真实姓名（可选）
   - 邮箱（可选，格式校验）
   - 手机号（可选，11 位数字）
   - 密码（必填，至少 6 位）
   - 所属组织（下拉选择，可选）
   - 角色分配（多选复选框）
4. 用户点击"保存"按钮
5. 系统校验表单数据，通过后调用 POST /api/users
6. 后端创建用户，记录审计日志（action="create"）
7. 前端显示成功提示，关闭对话框，刷新用户列表
8. 新用户在列表中显示，包含手机号和所属组织列

### 编辑用户

1. 用户在列表中点击某用户的"编辑"按钮
2. 系统弹出"编辑用户"对话框，加载用户详情（GET /api/users/{id}）
3. 表单回显当前数据：用户名（只读）、真实姓名、邮箱、手机号、密码（留空）、所属组织、角色
4. 用户修改字段后点击"保存"
5. 系统调用 PUT /api/users/{id}
6. 后端更新用户，记录审计日志（action="update"，beforeJson 为变更前数据）
7. 前端显示成功提示，关闭对话框，刷新列表

### 搜索用户

1. 用户在搜索框输入关键词
2. 前端防抖 300ms 后调用 GET /api/users?keyword=xxx&page=1
3. 后端返回匹配 username、real_name 或 email 的用户列表
4. 前端更新列表显示搜索结果
5. 用户清空搜索框时，前端重置 page=1 并重新加载全量列表

### 查看审计日志

1. 用户进入 `/admin/audit` 页面
2. 系统显示审计日志列表，默认显示所有模块
3. 用户选择 module 筛选器中的"user"选项
4. 系统过滤显示用户相关操作记录
5. 每条记录显示：操作时间、操作人、操作类型（create/update/delete）、目标用户、备注
6. 用户可点击展开查看详细变更（beforeJson / afterJson）

## 非功能性需求

- **性能**：搜索接口响应时间 < 500ms（keyword 模糊查询使用数据库 LIKE，数据量 < 10000 时可接受）
- **安全**：
  - 审计日志中密码字段必须脱敏
  - 用户只能查看自己租户的审计日志
  - 审计日志不可修改、不可删除
- **数据完整性**：
  - 手机号格式校验在前后端同时执行
  - groupId 必须指向当前租户存在的组织（或为 null）
- **向后兼容**：
  - 已有用户数据不受影响（phone 和 groupId 可为 null）
  - API 接口路径和权限控制不变（/api/users/*）
  - 前端路由不变（/users）

## 不做的事

- 不实现手机号验证码验证（仅格式校验）
- 不实现组织层级结构（当前为扁平组织）
- 不实现批量导入/导出用户
- 不实现用户头像上传（avatarUrl 字段保留但不在本次范围）
- 不改变现有 RBAC 权限模型
- 不实现审计日志导出功能

## 技术债务说明

1. **搜索性能**：当用户量超过 10000 时，LIKE 模糊查询性能下降，后续可考虑引入 Elasticsearch
2. **审计日志存储**：当前审计日志存储在数据库表中，长期运行后表会膨胀，后续需考虑归档策略
3. **组织关联**：当前用户只能属于一个组织（groupId），后续如需多组织支持需重构为多对多关系
