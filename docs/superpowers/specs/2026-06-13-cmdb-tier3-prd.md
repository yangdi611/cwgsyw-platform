# PRD: CMDB Tier 3 — 可视化（拓扑图增强 + JSONB diff + 变更统计 + 2D 视图）

> 前置：Tier 1（模型/属性/实例/关联/拓扑/变更历史）+ Tier 2（关联元数据/CSV 导入/影响分析）已审批通过。
> 本文档前半是 PRD（Product 负责），后半留给 Architect 写技术方案。

---

## 1. 背景与目标

Tier 1 + 2 让 CMDB 有了完整的 CRUD、关联、批量录入和影响分析能力，但日常运维还有三类痛点：

1. **变更历史"能查但不好看"**：Tier 1 的 `cmdb_change_log` 存了完整 `before_json` / `after_json`，但前端只显示原始 JSON 列表，字段多了根本看不出改了什么。需要 diff 高亮 + 时间线 + 过滤。
2. **拓扑图"能看但不实用"**：当前拓扑只展示节点和边，50+ 节点就糊成一团；没有按模型/状态着色、没有展开/折叠、没有过滤、没有对比——运维同学宁可去翻 Excel。
3. **缺少资产变动概览**：领导问"这个月资产变动了多少、哪些主机频繁改配置"，只能手跑 SQL。需要变更统计面板（今日/周/月 + Top 10 最活跃实例）。
4. **物理分布看不到**：机房的机柜/机架布局、主机在机柜中的位置，目前只能看列表；需要 2D 网格视图按 `idc` / `rack` 分组展示。

Tier 3 的目标：让 CMDB 从"能用"变成"好盯"——变更可追溯（diff 可视化）、拓扑可交互（展开/过滤/对比）、变动可度量（统计面板）、物理可布局（2D 视图）。

## 2. 用户故事

### 2.1 变更历史可视化（Enhanced Change History）

- 作为**运维工程师**，我想在实例详情页的"变更历史"tab 看到时间线视图（时间、操作人、动作类型、变更摘要），以便快速定位某次变更。
- 作为**运维工程师**，我想展开某条变更记录后看到 JSONB diff 高亮：红色=删除、绿色=新增、黄色=修改（before → after），以便一眼看出具体改了哪些字段。
- 作为**审计员**，我想按时间范围、操作人、动作类型过滤变更历史，以便追溯特定时间段内某个人的所有操作。
- 作为**审计员**，我想变更历史支持分页（默认 20 条/页，可切换 50/100），以便在大量记录中浏览。

### 2.2 变更统计面板（Change Statistics Dashboard）

- 作为**CMDB 管理员**，我想在 CMDB 首页看到"今日/本周/本月变更数"卡片（按创建/更新/删除拆分），以便汇报资产变动。
- 作为**CMDB 管理员**，我想点击统计卡片后跳转到变更历史列表页并自动带入时间范围过滤，以便深挖细节。
- 作为**CMDB 管理员**，我想看到"过去 30 天变更次数 Top 10 实例"，以便识别频繁配置漂移的资产。
- 作为**CMDB 管理员**，我想统计 API 支持按模型过滤（`?model_id=xxx`），以便查看某类资产（如"主机"）的变更趋势。

### 2.3 拓扑图可视化增强（Topology Visualization Enhancements）

- 作为**运维工程师**，我想拓扑图节点按模型类型着色（主机=蓝色、应用=绿色、数据库=橙色），以便一眼区分资产类型。
- 作为**运维工程师**，我想每个模型的颜色可以在"模型管理"页面自定义，以便匹配公司的视觉规范。
- 作为**运维工程师**，我想拓扑图节点按状态显示边框样式（online=实线绿色、offline=虚线红色、maintenance=虚线黄色），以便快速识别异常。
- 作为**运维工程师**，我想悬浮节点时显示详情卡片（名称、模型、状态、owner、`is_list_show=true` 的关键属性），以便不点进详情就能看概览。
- 作为**运维工程师**，我想点击节点后展开/折叠它的下游（递归），以便控制视觉复杂度。
- 作为**运维工程师**，我想在侧边栏按模型类型/状态/分组勾选过滤，未选中的节点灰显（不删除，保持拓扑连通性），以便聚焦关注范围。
- 作为**运维工程师**，我想使用"对比模式"：选择两个时间点（或某个变更单），系统渲染两个快照的拓扑差异（新增=绿色、删除=红色、修改=黄色），以便验证变更是否符合预期。
- 作为**运维工程师**，我想把拓扑图导出为 PNG（分辨率 ≥ 1920×1080），以便附在变更文档里。
- 作为**运维工程师**，我希望 500 节点 + 1000 边的拓扑图渲染不卡顿（前端虚拟化），以便应对大规模场景。

### 2.4 2D 视图（机房/机柜布局）

- 作为**运维工程师**，我想以网格形式查看某个模型的实例（如按 `idc` 分组看机房布局、按 `rack` 分组看机柜内主机），以便了解物理分布。
- 作为**运维工程师**，我想 2D 视图的每个单元格显示实例名称、状态、owner，并按状态着色，以便快速定位异常设备。
- 作为**运维工程师**，我想在 2D 视图顶部切换 `group_by` 属性（如从 `idc` 切到 `rack`），以便从不同维度观察分布。
- 作为**运维工程师**，我想点击 2D 视图的单元格跳转到实例详情页，以便查看具体配置。

## 3. 验收标准

### AC-1 变更历史可视化

- [ ] **AC-1.1** 实例详情页"变更历史"tab 渲染时间线视图，每条记录显示：时间、操作人、动作类型（创建/更新/删除）、变更摘要（如"修改了 3 个字段"）。
- [ ] **AC-1.2** 每条记录可展开查看完整 JSONB diff：红色=删除、绿色=新增、黄色=修改（before → after）。
- [ ] **AC-1.3** 变更历史支持按时间范围（最近 7 天 / 30 天 / 自定义）、操作人、动作类型过滤。
- [ ] **AC-1.4** 变更历史支持分页，默认 20 条/页，可切换 50/100。
- [ ] **AC-1.5** REST API `GET /api/cmdb/changes` 支持查询参数：`entity_type`、`entity_id`、`from`、`to`、`operator_id`、`action`、`page`、`size`；返回分页列表，每条含 `before_json`、`after_json`、`changed_fields[]`。

### AC-2 变更统计面板

- [ ] **AC-2.1** CMDB 首页新增"变更统计"卡片：今日/本周/本月变更数（按动作类型拆分：创建/更新/删除）。
- [ ] **AC-2.2** 统计卡片可点击跳转到变更历史列表页，自动带入时间范围过滤。
- [ ] **AC-2.3** REST API `GET /api/cmdb/changes/stats` 支持 `from`、`to`、`model_id` 参数；返回按日期聚合的变更统计（每天创建/更新/删除计数）。
- [ ] **AC-2.4** 统计面板展示"最活跃实例 Top 10"：过去 30 天变更次数最多的 10 个实例（含实例名称 + 变更次数）。
- [ ] **AC-2.5** 统计 API 响应加 Redis 缓存，TTL = 5 分钟；写入操作（实例 CRUD）后主动失效缓存。

### AC-3 拓扑图可视化增强

- [ ] **AC-3.1** `ci_model` 表新增 `color` 字段（VARCHAR(7)，如 `#1890FF`，默认值由系统分配）；拓扑图节点按模型类型着色。
- [ ] **AC-3.2** `ci_model` 表新增 `enable_2d_view` 字段（BOOLEAN，默认 false）；前端据此决定是否展示 2D 视图 tab。
- [ ] **AC-3.3** 拓扑图节点按状态显示边框：online=实线绿色、offline=虚线红色、maintenance=虚线黄色。
- [ ] **AC-3.4** 拓扑图节点悬浮显示详情卡片：名称、模型、状态、owner、`ci_attribute.is_list_show=true` 的关键属性。
- [ ] **AC-3.5** 拓扑图支持节点点击展开/折叠下游（递归）；展开后节点图标变为"−"，可再次点击折叠。
- [ ] **AC-3.6** 拓扑图侧边栏支持按模型类型/状态/分组勾选过滤；未选中节点灰显（opacity=0.3）但不删除，保持拓扑连通性。
- [ ] **AC-3.7** 拓扑图支持"对比模式"：选择两个时间点（或某个 `cmdb_change_log` 记录），渲染差异拓扑（新增节点=绿色、删除节点=红色、修改节点=黄色）。
- [ ] **AC-3.8** 拓扑图支持导出为 PNG（使用前端 svg-to-png 方案），分辨率 ≥ 1920×1080。
- [ ] **AC-3.9** 性能：500 节点 + 1000 边的拓扑图，前端 P95 渲染 < 2 秒（启用虚拟化或 canvas 渲染）。
- [ ] **AC-3.10** 模型管理页面可编辑 `color` 字段（颜色选择器），保存后立即反映到拓扑图。

### AC-4 2D 视图

- [ ] **AC-4.1** 实例列表页新增"2D 视图"tab，仅当该模型的 `ci_model.enable_2d_view=true` 时显示。
- [ ] **AC-4.2** 2D 视图以网格形式展示实例：每个单元格含名称、状态、owner，按状态着色（同 AC-3.3）。
- [ ] **AC-4.3** 2D 视图支持按某个属性分组（顶部下拉选择 `group_by` 属性），如按 `idc` / `rack` 分组。
- [ ] **AC-4.4** 点击单元格跳转到实例详情页。
- [ ] **AC-4.5** REST API `GET /api/cmdb/instances/2d-view` 支持 `model_id`、`group_by` 参数；返回按分组聚合的实例列表 `{ "group1": [...instances], "group2": [...] }`。

### AC-5 权限与审计

- [ ] **AC-5.1** 变更历史查询 API 受 `cmdb_change:read` 权限控制。
- [ ] **AC-5.2** 变更统计 API 受 `cmdb_change:read` 权限控制。

- [ ] **AC-5.5** 所有可视化 API 调用写入 `audit_log`（action=`view`，targetType=`cmdb_visualization` 或 `cmdb_change`）。
- [ ] **AC-5.6** 模型管理页面的 `color` 字段编辑受 `cmdb_model:update` 权限控制。

### AC-6 向后兼容

## 4. 交互流

### 4.1 变更历史可视化

```
用户进入"实例详情 → 变更历史"tab
  → 系统渲染时间线视图（按 created_at 倒序，默认展示最近 7 天）
  → 每条记录显示：时间、操作人、动作类型、变更摘要（如"修改了 3 个字段"）
  → 用户点击某条记录 → 展开 JSONB diff 视图
  → diff 视图：红色=删除、绿色=新增、黄色=修改（before → after）
  → 用户在侧边栏选择时间范围、操作人、动作类型过滤
  → 过滤后列表实时更新（前端 TanStack Query 重新拉取）
  → 用户切换分页 → 拉取下一页
```

### 4.2 变更统计面板

```
用户进入 CMDB 首页
  → 系统渲染"变更统计"卡片：今日/本周/本月变更数（按动作类型拆分）
  → 卡片下方展示"最活跃实例 Top 10"（实例名称 + 变更次数）
  → 用户点击卡片 → 跳转到变更历史列表页，自动带入时间范围过滤
  → 用户点击 Top 10 中的某个实例 → 跳转到该实例的变更历史
  → 用户可按模型过滤统计（下拉选择）→ 统计数据和 Top 10 更新
```

### 4.3 拓扑图可视化增强

```
用户进入"实例详情 → 拓扑图"tab
  → 系统渲染拓扑图（节点按模型类型着色、按状态显示边框）
  → 用户悬浮节点 → 显示详情卡片（名称、模型、状态、owner、关键属性）
  → 用户点击节点 → 展开/折叠下游（递归展开该节点的所有下游关系）
  → 用户在侧边栏勾选过滤条件 → 未选中的节点灰显（opacity=0.3）
  → 用户点击"对比模式" → 选择两个时间点（或某个变更单）
  → 系统渲染差异拓扑（新增=绿、删除=红、修改=黄、未变=灰）
  → 用户点击"导出 PNG" → 下载 1920×1080 拓扑截图
```

### 4.4 2D 视图

```
用户进入"模型 X 的实例列表"页
  → 若 ci_model.enable_2d_view=true → 显示"2D 视图"tab
  → 用户切换到"2D 视图"tab
  → 系统渲染网格视图（默认按第一个 string 类型属性分组）
  → 每个单元格代表一个实例，显示名称、状态、owner，颜色按状态着色
  → 用户在顶部下拉切换 group_by 属性 → 视图重新渲染
  → 用户点击单元格 → 跳转到实例详情页
```

## 5. 非功能性需求

### 5.1 性能

- 变更历史分页查询 P95 < 300ms（100k 条记录规模，复合索引 `(entity_type, entity_id, created_at)`）。
- 变更统计 API P95 < 500ms（按日期聚合 30 天范围，走 Redis 缓存）。
- 拓扑图渲染 500 节点 + 1000 边：前端 P95 < 2 秒（启用虚拟化或 canvas 渲染）。
- 2D 视图渲染 1000 个单元格：前端 P95 < 1 秒。

### 5.2 安全

- 所有可视化 API 遵循现有 RBAC：`cmdb_change:read` 或 `cmdb_instance:read`。
- 拓扑图对比模式仅返回用户有权限查看的节点（越权节点在差异拓扑中标记为"已隐藏"）。
- 变更统计 API 不暴露敏感字段（仅返回计数，不返回具体字段值）。

### 5.3 可靠性

- 变更历史查询加索引：`cmdb_change_log` 表按 `(entity_type, entity_id, created_at)` 建复合索引（如 Tier 1 已建则复用）。
- 变更统计 API 加 Redis 缓存（TTL=5 分钟），写入操作后主动失效。
- 拓扑图对比模式加超时（默认 5 秒），超时返回"无法生成对比"提示。

### 5.4 可观测

- 变更历史查询打 metric：`cmdb.changes.query.duration_seconds`、`cmdb.changes.query.count`。
- 变更统计 API 打 metric：`cmdb.changes.stats.duration_seconds`。
- 拓扑图渲染打 metric（前端）：`cmdb.topology.render.duration_ms`、`cmdb.topology.node_count`。

## 6. 不在范围（明确排除）

- ❌ 自动发现（agent 主动采集主机/应用信息）→ Phase 4。
- ❌ 与外部监控系统集成（Prometheus、Zabbix）→ Phase 4。
- ❌ 变更历史的归档/清理策略 → Phase 4。
- ❌ 拓扑图的 3D 渲染 → 不做。
- ❌ 2D 视图的拖拽编辑（手动调整实例在网格中的位置）→ 不做，2D 视图仅用于展示。
- ❌ 变更历史的审批流（"申请回滚某次变更"走 Flowable）→ 不做，变更历史仅读。
- ❌ 拓扑图的实时推送（WebSocket 推送节点变化）→ 不做，用户手动刷新。

## 7. 开放问题（留给 Architect / 评审）

1. **变更历史 diff 的存储优化**：当前 `before_json` / `after_json` 存完整 JSONB；是否改为仅存 `changed_fields` 的 diff（减存储）？还是保留完整快照以便未来扩展？
2. **拓扑图对比模式的数据源**：从 `cmdb_change_log` 反推历史拓扑快照（简单但查询复杂）？还是新增 `cmdb_topology_snapshot` 表定期快照（占存储但查询快）？
3. **2D 视图的 group_by 属性范围**：仅限 `field_type=string` 的属性？还是允许 enum、reference 类型？
4. **变更统计的缓存策略**：按用户缓存（不同用户看到的统计因权限不同）？还是全局缓存（统计结果不区分权限，前端过滤）？
5. **拓扑图导出的分辨率**：PNG 导出是否支持自定义分辨率（如 4K）？还是固定 1920×1080？
6. **拓扑图对比的时间点选择**：是否支持"选择任意历史时间戳"？还是仅支持"选择某个变更单"（简化实现）？
