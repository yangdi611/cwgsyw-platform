# IT 运维运营平台 — 系统设计文档

**日期：** 2026-05-21  
**状态：** 已确认，待实施  
**目标：** 从单团队 MVP 出发，架构预留多租户 SaaS 升级路径

---

## 1. 项目定位

面向 IT 基础设施运维团队的内部工具平台，整合工作报告、变更管理、文档存储、设备密码管理四大核心业务，配备可配置审批流和动态权限管理。

当前阶段服务单个组织（5个组，约数十人），架构从第一天起预留多租户扩展能力，目标是分阶段演进至 SaaS。

## 3. 组织结构

```
超级管理员（平台级，跨租户）
└── 管理组（管理员，租户内最高权限，可多人）
    ├── 数据库组（组长 N 人 + 组员）
    ├── 主机组（组长 N 人 + 组员）
    ├── 网络组（组长 N 人 + 组员）
    └── 云平台组（组长 N 人 + 组员）
```

- 每组可有多名组长，审批采用**或签**（任意一人通过即生效）
- 用户可同时属于多个角色，权限取并集
- 文档管理员是独立角色，可叠加到任何用户上

## 4. 动态权限系统（RBAC）

### 4.1 数据模型

```
Resource（资源）        — 模块名 + 操作列表（如 change_doc: [create, read, approve, export]）
    ↓
Permission（权限）      — resource + action 的组合（如 change_doc:approve）
    ↓
Role（角色）            — 关联 N 个 Permission，含 scope 字段
    ↓
User（用户）            — 关联 N 个 Role
```

### 4.2 角色 scope

- `group`：权限范围限定在用户所属组
- `tenant`：租户内全局权限
- `platform`：跨租户平台级权限（超级管理员专用）

### 4.3 初始角色定义

| 角色 | Scope | 说明 |
|------|-------|------|
| 超级管理员 | platform | 平台全权限，管理所有租户 |
| 管理员 | tenant | 租户内全权限，配置审批流/AI/水印 |
| 运维组长 | group | 审批本组单据，管理本组密码库 |
| 运维组员 | group | 填报日报/变更，按分类查看密码 |
| 文档管理员 | tenant | 专职管理共享文档，不参与运维流程 |

### 4.4 新增模块时的强制检查（写入 CLAUDE.md）

每次新增或修改业务模块时，必须完成以下四步才算完成：

1. 在 `sys_resource` 表注册新资源名称和操作列表
2. 确认前端权限配置页正确渲染新权限项（由后端接口动态返回）
3. 确认后端所有新接口都有 `@PreAuthorize` 注解覆盖
4. 默认所有角色对新模块无权限（最小权限原则），由超级管理员手动分配

## 5. 核心模块设计

### 5.1 工作报告系统

**流程：** 填写日报 → 提交 → 组长审批（或签）→ 归档 → 月报聚合 → 季报导出

**日报表单字段：**
- 今日完成事项（结构化列表）
- 遇到的问题及处理结果
- 明日工作计划
- 工时记录（可选）

**提醒机制：** 系统在可配置时间（默认下班前 30 分钟）通过邮件和微信公众号模板消息推送提醒，消息含直接填写链接。

**月报/季报：** 自动聚合所选时间段内已审批日报数据，支持按组/按人导出 Excel 和 PDF。

**数据留存：** 所有日报提交记录、审批操作、修改历史全部写入数据库，不可物理删除，仅支持逻辑归档。

### 5.2 变更文档系统

**流程：** 填写变更信息 → AI 辅助生成内容 → 人工审阅修改 → 提交审批 → 审批通过 → 生成邮件内容 + 导出文档

**两份固定文档模板：**

1. **变更申请单**：变更编号（自动生成+可覆盖，格式 `CHG-YYYY-MMDD-NNN`）、申请人、申请时间、变更内容描述、影响范围、变更时间窗口、资源支持说明、审批签字区
2. **变更方案**：背景与目的、详细操作步骤、风险评估与应对措施、回滚计划、验证方法、相关人员联系方式

**AI 辅助：** 用户填写基本信息后，调用配置的大模型 API 自动生成操作步骤、风险评估、回滚计划等内容，用户可在富文本编辑器中修改。

**模板管理：**
- Word 模板（.docx）作为导出基础，通过书签占位符填充数据
- 在线富文本编辑器用于预览和字段映射调整
- 模板存储在数据库中，支持版本历史

**邮件生成：** 审批通过后，系统根据可配置邮件模板（含变量占位符）生成邮件正文和收件人列表，供用户复制后手动发送，不由系统直接发送。

**水印配置：** 文字内容、字体大小、角度、透明度、署名（公司名/部门名/LOGO）均可在系统设置中配置。

**跨组变更：** 变更单可关联多个组，审批流可配置多组组长会签。

**数据留存：** 变更单每次修改保存完整快照，审批每个节点的操作人、时间、意见全部记录，导出的文件引用存储在 MinIO 并记录到数据库。

### 5.3 共享文档存储

- 分类目录结构，文档管理员管理目录和上传
- 所有用户可按权限查看，支持在线预览（PDF、图片、Office 文件）
- 文件版本历史：每次更新保留旧版本，可回滚
- 文件元数据（上传人、时间、大小、版本）存数据库，文件实体存 MinIO
- 不可物理删除，仅逻辑删除并保留记录

### 5.4 设备密码库

- 设备信息（名称、IP、类型、所属组、备注）+ 账号密码列表
- 密码字段使用 AES-256 加密存储，密钥通过环境变量注入，不入库
- 查看密码需要相应权限，组员按分类查看（由组长配置可见分类）
- 每次查看密码的操作（用户、时间、设备、IP）全部写入审计日志
- 支持按组、按设备类型分类检索

### 5.5 AI 网关

- 统一接口适配器，后端屏蔽各模型 API 差异
- 支持：Kimi（月之暗面）、DeepSeek、智谱 GLM
- 管理员页面可配置：启用的模型、API Key、默认模型、每次调用的 system prompt 模板
- API Key 加密存储，不在前端返回明文
- 调用记录（模型、token 消耗、耗时、成功/失败）写入数据库

## 6. 审批流引擎（Flowable）

- 基于 BPMN 2.0 标准，管理员通过页面化配置器定义审批流
- 支持节点类型：顺序审批、并行审批（会签）、或签、条件分支
- 或签逻辑：同一节点多个审批人，任意一人操作即推进
- 适用单据类型：日报、变更文档、共享文档（需审批时）
- 每个审批节点可绑定角色或指定人员
- 审批操作（通过/拒绝/转交）、操作人、时间、意见全部持久化

---

## 7. 审计与历史数据

所有业务数据遵循以下原则：

- **不可物理删除**：业务数据只做逻辑删除（`is_deleted` + `deleted_at` + `deleted_by`）
- **操作审计表**：独立的 `audit_log` 表记录所有写操作（模块、操作类型、操作人、时间、变更前后快照 JSON）
- **密码查看审计**：单独的 `password_access_log` 表，记录每次查看
- **文档版本历史**：变更文档和共享文档每次修改保存完整内容快照
- **审批流水**：Flowable 引擎自身持久化所有流程实例和历史，不额外删除

## 8. 通知中心

| 通知渠道 | 使用场景 |
|---------|---------|
| 邮件（SMTP） | 日报提醒、审批待办、审批结果通知 |
| 微信公众号模板消息 | 日报提醒、审批待办（需认证服务号） |
| 站内消息 | 所有通知均同步站内，可在平台内查看历史 |

通知配置（提醒时间、接收渠道、模板内容）由管理员在系统设置中配置。

## 9. 多租户升级路径

| 阶段 | 状态 | 内容 |
|------|------|------|
| 阶段一（当前） | 开发中 | 所有表含 `tenant_id`，值固定为 `default`；单套部署 |
| 阶段二 | 未来 | 启用 `tenant_id` 路由中间件；添加租户注册/管理页；每个租户独立配置 |
| 阶段三 SaaS | 未来 | 计费模块接入；可选独立数据库部署；运营后台 |

## 10. 部署架构

```
Docker Compose 服务清单：
├── nginx          — 反向代理 + 前端静态文件
├── frontend       — Next.js 应用（Node.js）
├── backend        — Spring Boot JAR
├── postgres       — PostgreSQL 16
├── redis          — Redis 7
└── minio          — MinIO 对象存储
```

所有服务配置通过环境变量注入（`.env` 文件），不硬编码敏感信息。
