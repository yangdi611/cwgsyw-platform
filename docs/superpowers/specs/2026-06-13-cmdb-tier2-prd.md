# PRD: CMDB Tier 2 — 关联增强 + CSV 导入 + 影响分析

> 前置：Tier 1 已上线（模型/属性/实例/关联/拓扑），PRD 见 `specs/2026-06-12-cmdb-tier1-prd.md`（仓库历史）。
> 本文档前半是 PRD（Product 负责），后半留给 Architect 写技术方案。

---

## 1. 背景与目标

Tier 1 建立了 CMDB 的骨架：模型定义 → 属性 Schema → 实例 CRUD → 关联关系 → 拓扑图。但实际运维场景中，运维同学还缺三块能力：

1. **关联本身需要携带信息**：比如 "主机-运行-应用" 这条关系，要记录监听端口、部署日期、负责人；现在 `ci_instance_rel` 只有 kind，没有扩展字段。
2. **批量录入效率太低**：机房搬迁、资产盘点一次几百上千条主机，目前只能一条条录入；需要 CSV 导入 + 冲突处理。
3. **变更/故障时看不到影响面**：一台宿主机下线前，应该能自动列出"上面跑了哪些应用、应用又依赖哪些下游"，否则变更评审没有数据支撑。

Tier 2 的目标：让 CMDB 从"能看"变成"能用"——关联可描述、数据可批量灌入、数据可反向推演影响。

## 2. 用户故事

### 2.1 关联关系增强（Relationship Metadata）

- 作为**运维工程师**，我想在"主机 → 运行 → 应用"的关联上记录监听端口、部署日期、备注，以便变更评审时能看到具体配置。
- 作为**架构师**，我想定义"关联类型上有哪些扩展属性"（每个 `association_kind` 可配置自己的 schema），以便不同类型的关系携带不同的信息。
- 作为**审计员**，我想在拓扑图上看到关联的元数据（悬浮显示），以便快速判断关系的有效性。
- 作为**管理员**，我想批量给同一类关联补字段（比如一次性给所有 "run" 关系补上部署日期），以便历史数据回填。

### 2.2 CSV 批量导入

- 作为**运维工程师**，我想从 Excel 导出 CSV 后直接上传到 CMDB，按模型选择 → 列映射 → 预览 → 确认导入，以便一次性录入几百台主机。
- 作为**管理员**，我想在导入预览中看到"新增 N 条 / 更新 M 条 / 跳过 K 条 / 失败 J 条（带原因）"，以便在确认前核对。
- 作为**运维工程师**，我想指定"唯一键"（例如 hostname 或 inner_ip）来决定是新增还是更新已有实例，以便避免重复录入。
- 作为**审计员**，我想导入完成后能查到"哪次导入、谁操作、每条数据的 before/after"，以便追溯。
- 作为**运维工程师**，我想下载当前模型下所有实例的 CSV 作为导入模板，以便字段名和格式对得上。
- 作为**运维工程师**，我想上传后如果某几行校验失败，能下载"失败行 + 原因"的 CSV，以便修正后重传。

### 2.3 影响分析（Impact Analysis）

- 作为**变更申请人**，在我勾选一台主机准备下线前，我想点"影响分析"按钮，看到"直接依赖这条主机的应用列表 + 这些应用的下游依赖"，以便判断变更风险。
- 作为**审批人**，我想在审批变更单时看到影响分析的拓扑子图（高亮受影响节点），以便决定是否需要通知业务方。
- 作为**运维工程师**，我想指定"分析深度"（1 跳 / 2 跳 / 3 跳）和"方向"（只查下游 / 只查上游 / 双向），以便控制影响面的范围。
- 作为**运维工程师**，我想把影响分析的结果导出成 PDF/截图，以便附在变更文档里。
- 作为**审批人**，我想看到影响分析结果里每个节点的"业务等级"（来自实例的某个字段），以便优先关注关键业务。

## 3. 验收标准

### AC-1 关联元数据

- [ ] **AC-1.1** 管理员可在"模型管理 → 关联定义"页面为任意 `association_kind` 添加扩展属性（字段名、类型、是否必填），存储到新增的 `ci_association_attr_def` 表。
- [ ] **AC-1.2** 用户在创建/编辑关联时，可填写该 kind 的扩展字段；校验规则与实例属性一致（类型、必填、枚举值范围）。
- [ ] **AC-1.3** 拓扑图节点间的连线悬浮后，显示该关联的所有元数据（key: value 列表）。
- [ ] **AC-1.4** 关联的元数据修改纳入审计日志（`ci_instance_rel` 的 before/after 已包含，无需新增机制）。
- [ ] **AC-1.5** 内置 5 个 `association_kind` 默认为"无扩展属性"，向后兼容；现有实例关系无需迁移即可正常显示。

### AC-2 CSV 导入

- [ ] **AC-2.1** 用户在"实例列表 → 导入"按钮，选择 CSV 文件（UTF-8 / GBK 自动识别），系统解析后返回预览。
- [ ] **AC-2.2** 预览页显示：总行数、待新增、待更新、待跳过、校验失败；失败行可展开查看具体原因（字段类型错、必填缺失、唯一键冲突等）。
- [ ] **AC-2.3** 用户可下载"失败行 + 原因"的 CSV，修正后重新上传。
- [ ] **AC-2.4** 用户选择"冲突策略"：按唯一键覆盖 / 跳过已存在 / 报错中断；默认为"按唯一键覆盖"。
- [ ] **AC-2.5** 一次导入上限 5000 行；超过时提示分批。单文件大小上限 10MB。
- [ ] **AC-2.6** 导入结果写入 `audit_log`：每条新增/更新的实例各自一条记录，`remark` 字段带 `batch_id=<uuid>`，可聚合成"同一次导入"。
- [ ] **AC-2.7** 提供"下载模板"按钮：导出当前模型所有属性列（含 enum 选项说明）的空 CSV，作为导入样板。
- [ ] **AC-2.8** 枚举字段在 CSV 中按字面值（如 `online`）填写；导入时自动校验是否在 `enum_options` 范围内。
- [ ] **AC-2.9** 导入任务为同步操作，超过 30 秒时显示进度条（已处理 N/总数 M）；前端轮询后端进度接口。

### AC-3 影响分析

- [ ] **AC-3.1** 实例详情页增加"影响分析"按钮；点击后弹窗选择方向（上游/下游/双向）和深度（1~5 跳），返回影响清单。
- [ ] **AC-3.2** 影响清单按"跳数"分层展示：第 1 跳直接关联、第 2 跳间接关联……每层显示节点名称、模型、状态、关联路径（A → B → C）。
- [ ] **AC-3.3** 影响分析结果可渲染为拓扑子图（复用现有拓扑渲染组件，高亮起点和路径）。
- [ ] **AC-3.4** 性能：深度 3 跳、涉及 500 个节点的分析在 3 秒内返回；超过 5 跳或节点 >2000 时，后端截断并在响应里标注 `truncated=true`。
- [ ] **AC-3.5** 提供 REST API `POST /api/cmdb/instances/{id}/impact`，变更单模块可调用（为未来 Phase 3 变更文档集成预留）。
- [ ] **AC-3.6** 影响分析结果支持"导出为图片"（PNG），使用前端 svg-to-png 库生成。
- [ ] **AC-3.7** 已软删除的实例不参与影响分析路径计算。

## 4. 交互流

### 4.1 关联元数据编辑流

```
用户进入"实例详情 → 关联"tab
  → 点击某条关联的"编辑"按钮
  → 弹窗显示：源实例（只读）、目标实例（只读）、关联类型（只读）、扩展属性表单（动态根据 association_kind 的 attr_def 渲染）
  → 填写/修改扩展字段
  → 提交 → 后端校验类型与必填 → 写 audit_log → 刷新列表
```

### 4.2 CSV 导入流

```
用户在"模型 X 的实例列表"页
  → 点击"导入"按钮
  → 弹窗第一步：上传 CSV + 选择冲突策略（覆盖/跳过/报错）
  → 后端解析：自动识别编码，按第一行当表头，校验字段类型/必填/枚举/唯一键
  → 弹窗第二步（预览）：显示统计 + 失败行明细；用户可"下载失败 CSV"
  → 用户确认"执行导入"
  → 后端在单事务内批量写入（每 100 条一个事务分段，避免长事务）；前端轮询进度
  → 完成后弹窗第三步（结果）：成功 N / 更新 M / 跳过 K / 失败 J；可下载本次导入的完整记录 CSV
```

### 4.3 影响分析流

```
用户在"实例详情"页
  → 点击"影响分析"按钮
  → 弹窗：方向（上游/下游/双向）+ 深度（1~5）
  → 提交 → 后端 BFS 遍历 `ci_instance_rel`，按跳数收集节点
  → 返回分层结果（list 形式 + 拓扑 subgraph）
  → 用户在弹窗内查看清单；可切换"清单视图 ↔ 拓扑图视图"
  → 可点击"导出图片" → 下载 PNG
  → 可点击"用于变更单" → 跳转变更单创建页，自动带入受影响实例 ID 列表（Phase 3 集成）
```

## 5. 非功能性需求

### 5.1 性能
- CSV 导入 5000 行 / 10MB：端到端 < 60 秒（含解析、校验、入库、审计）。
- 影响分析 3 跳 / 500 节点：< 3 秒 P95。
- 拓扑图渲染 1000 节点 + 2000 边：前端不卡顿（启用虚拟化）。

### 5.2 安全
- CSV 导入遵循现有 RBAC：`cmdb:instance:create` 和 `cmdb:instance:update` 都校验；用户只能导入自己有权限的模型。
- 影响分析只返回用户有 `cmdb:instance:read` 权限的节点（越权的节点在路径中标记为"已隐藏"，不暴露名称，但路径仍连通——避免拓扑信息泄露）。
- CSV 文件上传后在后端临时目录处理，处理完即删除，不持久化。

### 5.3 可靠性
- CSV 导入分段事务：每段失败不影响其他段；失败段记录详细错误，整批不中断。
- 影响分析加超时（默认 5 秒），超时返回已收集到的部分结果 + `truncated=true`。

### 5.4 向后兼容
- 现有 `ci_instance_rel` 数据无需迁移即可保留；新增的 `metadata` JSONB 字段默认为空 `{}`。
- 现有拓扑图、实例列表、关联列表在 Tier 2 上线后行为不变。

### 5.5 可观测
- 导入任务打 metric：`cmdb.import.total`, `cmdb.import.duration_seconds`, `cmdb.import.rows_failed`。
- 影响分析打 metric：`cmdb.impact.duration_seconds`, `cmdb.impact.depth`, `cmdb.impact.node_count`。

## 6. 不在范围（明确排除）

- ❌ 自动发现（agent 主动采集主机/应用信息）→ Phase 4 考虑。
- ❌ 与外部 CMDB（如 iTop、蓝盾）双向同步 → Phase 4。
- ❌ 关联的审批流（"申请新建关联"走 Flowable）→ 不在这期，关联仍是有权限即可建。
- ❌ CSV 导出全部实例（只导出"导入模板"即空表头 + 枚举说明；全量导出放 Phase 4 报表）。
- ❌ 影响分析结果直接触发变更单 → 只预留跳转入口，实际集成在 Phase 3 变更单重构。

## 7. 开放问题（留给 Architect / 评审）

1. **关联元数据存储选型**：`ci_instance_rel` 加 `metadata JSONB` vs 新建 `ci_instance_rel_attr` 行存表？前者简单但查询不便，后者查询灵活但 JOIN 多。
2. **CSV 导入的"唯一键"范围**：仅限模型 Schema 中 `is_unique=true` 的属性？还是允许用户在导入时临时指定组合唯一键（如 `hostname + idc`）？
3. **影响分析的权限模型**：是"路径连通但节点信息屏蔽"（当前 PRD），还是"直接不返回越权节点"（更严格但拓扑断裂）？
4. **导入进度的轮询 vs WebSocket**：当前 PRD 写轮询，但如果 Phase 4 还要加"导出长任务"，是否统一升级到 WebSocket/SSE？
5. **CSV 编码识别的兜底**：自动识别失败时是否让用户手动选（UTF-8 / GBK / BIG5）？
