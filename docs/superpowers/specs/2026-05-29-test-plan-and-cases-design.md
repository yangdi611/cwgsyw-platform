# 全面精细化浏览器测试方案与用例 — 设计规格

## 目标

从客户/用户视角，为 cwgsyw-platform 全部 10 个功能模块创建完整的手工测试套件，覆盖正常流程、异常边界、权限隔离和 UI 交互。

## 产出物

```
test/
├── README.md              # 目录说明与使用指引
├── 测试方案.md             # 测试策略、环境、账号、覆盖矩阵
├── 测试用例.md             # 详细用例（约 100 条，P0 步骤级 + P1/P2 场景级）
└── fixtures/              # 测试数据文件
    ├── sample.xlsx
    ├── sample.docx
    ├── sample.pdf
    └── sample.png
```

`test/` 已加入 `.gitignore`，不上传 GitHub。

## 测试方案.md 大纲

1. **测试目标与范围** — 本轮覆盖的 10 个模块，不含自动化执行
2. **测试环境** — docker-compose 本地六容器，地址端口，启动命令
3. **测试账号矩阵** — 5 角色 ×（账号/密码/角色/scope/权限清单）
4. **模块覆盖矩阵** — 10 模块 × 4 维度打勾表
5. **测试优先级定义** — P0(阻塞)/P1(严重)/P2(一般)/P3(建议)
6. **测试执行流程** — 接任务→执行→记录→提bug
7. **缺陷追踪** — 关联 GitHub Issues，字段规范
8. **测试报告模板** — 执行汇总格式

### 用例子分类

每个模块按以下维度组织：
- **正常流程**（P0 步骤级）— 用户最常走的 Happy Path
- **异常/边界**（P1 场景级）— 空值、超长、非法输入
- **权限隔离**（P1 场景级）— 越权访问、跨组不可见
- **UI/交互**（P2 场景级）— 分页、排序、加载状态

## 设计决策

- **仅本地环境**：所有用例基于 `docker compose up -d` 后的 `localhost:3000`
- **纯手工文档**：不包含 Playwright 代码、YAML 自动化脚本（后续阶段补充）
- **按模块组织**：每个模块独立章节，测试人员可按需聚焦
- **P0 写死步骤**：确保即使不熟悉系统的人也能执行
- **前置条件明确**：含测试数据准备指引，减少"测不了"的情况
- **步骤级含输入值**：如 `输入 Admin@123`，不是 `输入密码`

## 不覆盖

- 自动化测试脚本（Playwright MCP YAML、API 测试）
- 性能/压力测试
- 移动端/响应式测试
- 多租户 SaaS 场景（当前 MVP 单租户）
