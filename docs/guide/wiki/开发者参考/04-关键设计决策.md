# 关键设计决策

实施前经代码核实拍板的 4 项决策（完整依据见 `docs/spec/spec.md` 第 25 节）。

## 决策 1 — 正文存 DB，附件存 MinIO（混合）

Markdown 正文存 PostgreSQL `TEXT`，图片/附件存 MinIO。

**理由**：正文典型 1KB–100KB，`TEXT` 足够；DB 单库事务保证一致性，全文搜索可直接 ILIKE。二进制附件天然适合对象存储，复用 SharedFile 模式。业界（Confluence/GitLab Wiki）同此。

## 决策 2 — 搜索用 ILIKE，不建 FTS 索引

`title/content ILIKE '%kw%'`，摘要用 `substring`。

**理由**：`to_tsvector('simple', 中文)` 无分词能力，命中实际全靠 ILIKE；与 CMDB 搜索一致；内部团队体量下全表扫描足够。建一个对中文无效的 GIN 索引只增维护成本。未来需要时引入 `pg_jieba`。

## 决策 3 — 附件后端流式代理，不用预签名 URL

`GET /api/wiki/attachments/{id}` 后端从 MinIO `download` 后 `transferTo` 响应流。

**依据（代码核实）**：`minio.endpoint=http://minio:9000` 是 Docker 内网主机名，浏览器不可达；nginx 未代理 MinIO；预签名 URL 会带 `minio:9000` 域名无法加载。图片是 Markdown 内自动加载的 `<img>`，必须无缝可达 → 只有后端代理在 dev/prod 都工作。

## 决策 4 — 审批流程线上配置，BPMN XML 不入仓库

`wiki_publish` 流程由管理员在 `/workflow/admin` 在线配置，`WikiPublishListener`（Java）随代码部署。

**理由**：平台已有 BPMN 在线编辑能力；放文件改流程要 rebuild backend，在线配置零部署；与「日报审批」管理方式统一。

## 实现期发现并修正的问题

| 问题 | 根因 | 修正 |
|------|------|------|
| 前端权限守卫失效 | UI 用 `hasPermission('wiki','write')`，但全局动词是 `update`（`write` 仅 ACL 内部动词） | 3 处 `write`→`update` |
| 删除页面/空间不生效（返回 200 但 is_deleted 仍 false） | `@TableLogic` 字段被 MyBatis-Plus 从 `updateById` 中剥离，手动 set 无效 | 改用 `deleteById` |

两者均经 API 验收回归确认修复。
