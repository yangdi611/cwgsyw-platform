# 页面 ACL 配置

页面级访问控制（ACL）采用 **NTFS 风格的覆盖式继承**，与共享文件库的文件夹 ACL 同一套设计。

## 入口

页面阅读视图右侧 **权限** 面板 / ACL 对话框（需 `wiki:manage_acl` 权限，🔒 图标据此显示）。

## 权限动词

ACL 条目的权限是以下 4 个动词的组合：

| 动词 | 含义 |
|------|------|
| `read` | 阅读页面内容 |
| `write` | 编辑内容、新建子页 |
| `delete` | 删除页面 |
| `publish` | 直接发布 |

## 授权对象（subject）

每条 ACL 指向一个授权对象，三种类型：

| subject_type | 含义 | 匹配方式 |
|--------------|------|---------|
| `role` | 按角色 | 用户拥有该角色即匹配 |
| `group` | 按用户组 | 用户所属组匹配 |
| `user` | 按具体用户 | 用户 ID 精确匹配 |

## 覆盖式继承（核心语义）

- 每个页面有 `acl_inherited` 标志：
  - `true`（默认）：继承父页面的 ACL
  - `false`：使用本页自定义 ACL
- **生效规则**：从当前页自下而上沿父链查找，**第一个 `acl_inherited=false` 的祖先**，用它的 ACL 行。
- 若整条链到根都是继承（无任何自定义 ACL）→ 视为无 ACL 限制，仅受全局 RBAC 控制（向后兼容）。

### 覆盖能力

子页面设为「自定义」后，可比父级**更严**（移除条目）或**更松**（增加条目），且互不影响兄弟节点。

## 配置示例

「网络架构」空间下，根页面对全员可读，但「核心交换配置」子页面只对网络组可读写：

1. 打开「核心交换配置」页面的 ACL 对话框
2. 关闭「继承父级」开关（`inherited=false`）
3. 添加条目：
   - 类型 `group`，选「网络组」，勾选 `read` + `write`
4. 保存

此后只有网络组（及 admin）能读写该页，其它人即便对父页有权限也被拦截。

## 验收确认

API 验收已确认：默认 `inherited=true` 且 entries 为空；设为自定义（role + read）后再读取，正确返回 `inherited=false` 并解析出角色名称（如「运维组员」）。

## 审计

每次 ACL 变更写审计日志：`module=wiki`, `action=acl_update`, `targetType=wiki_page`，含变更前后快照。
