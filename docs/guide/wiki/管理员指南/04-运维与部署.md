# 运维与部署

## 附件存储

- Wiki 页面内嵌的图片/附件存于 **MinIO**，复用 `shared_file` 表（`source_type='wiki_page'`，`source_id=页面ID`）
- MinIO 对象 key 约定：`wiki/<spaceId>/attachments/<uuid>.<ext>`
- 正文中图片引用为 `/api/wiki/attachments/<fileId>`

### 为何用后端流式代理而非预签名 URL

平台 MinIO endpoint 为 Docker 内网 `http://minio:9000`，浏览器无法解析，nginx 也未代理 MinIO。因此附件读取走 **后端流式代理**（`GET /api/wiki/attachments/{id}` → 后端从 MinIO 下载并 `transferTo` 响应流），在 dev/prod 都可用，无需暴露 MinIO。

响应头：`Content-Type` 按扩展名推断，`Cache-Control: private, max-age=86400`（对象不可变，可长缓存）。

## 导出超时

批量导出整个空间是同步操作。若空间页面/图片很多导致慢，参考备份模块为导出端点放宽 nginx 超时：

```nginx
location ~ ^/api/wiki/.*/export {
    proxy_pass http://backend:8080;
    proxy_read_timeout 300s;
    proxy_send_timeout 300s;
}
```

改后 `docker compose -f docker-compose.dev.yml restart nginx`（nginx 用 bind mount，无需 rebuild）。

## 附件上传大小

复用平台现有 `spring.servlet.multipart.max-file-size/max-request-size`（备份模块已设 2GB），无需额外调整。

## 审计日志

所有写操作写 `audit_log`，`module='wiki'`：

| action | 触发 |
|--------|------|
| `create` | 创建空间/页面 |
| `update` | 保存页面 |
| `delete` | 删除空间/页面 |
| `move` | 移动页面 |
| `publish` | 直接发布 |
| `submit_review` | 提交审批 |
| `publish_approved` / `publish_rejected` | 审批完成（operatorId=0 系统） |
| `acl_update` | 修改页面 ACL |

## 数据库

- 5 张表：`wiki_space` / `wiki_page` / `wiki_page_version` / `wiki_backlink` / `wiki_page_acl`
- 均含 `tenant_id`（多租户预留，当前固定 `default`）
- `wiki_space` / `wiki_page` / `wiki_page_acl` 走 `@TableLogic` 逻辑删除（**删除必须用 `deleteById`**，手动 set `isDeleted` 会被 MyBatis-Plus 忽略）
- `wiki_page_version` / `wiki_backlink` 无逻辑删除（版本永久保留；反向链接随保存重建）

## 构建与迁移

```bash
# 后端（--no-cache 确保 V53 进 JAR）
docker compose -f docker-compose.dev.yml build --no-cache backend
docker compose -f docker-compose.dev.yml up -d backend
# 验证迁移
docker compose -f docker-compose.dev.yml logs backend | grep "version \"53"
# 前端
docker compose -f docker-compose.dev.yml build frontend
docker compose -f docker-compose.dev.yml up -d frontend
```
