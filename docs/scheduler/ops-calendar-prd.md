## 修订记录

### 2026-06-30 评审决策

1. **日报不降级**：日报数据是周/月/季/半年报的基础数据，`daily_report` 保留为一等任务类型并纳入素材归集。生成策略为「每天一条汇总跟踪任务 + 向未提交者扇出通知」，既不降级、又不让每人每天一条可闭环任务淹没日历。详见技术规格 9.2。
2. **工作台卡片替换**：确认用「运维日历」卡片替换主工作区「近期 CMDB 变更」卡片（已拍板）。
3. **节假日相对触发（holiday_relative）推迟到 Phase 4**：MVP 与 Phase 2 不做工作日相对推算；客户 8 类需求先用 quarterly/weekly/monthly + 手工节假日条目或固定日期任务覆盖。Phase 2 可维护节假日和排班，真正的“节前/节后第 N 个工作日”自动推算在 Phase 4 落地。
4. **审计日志**：所有写操作（规则/任务/排班/节假日的增删改、任务状态流转）必须在同一事务内写平台 `audit_log`（项目强制规则 2），与领域表 `ops_schedule_task_log` 并存。
5. **调度器不全表扫描**：生成阶段按 `next_generate_at <= now` 过滤，命中 `idx_ops_rule_next_generate` 索引，生成后回写；不遍历所有 enabled 规则。
6. **通知模板渲染 null-safe**：`assigneePhone` 等字段可能为空（用户管理功能后续完善），渲染时缺失变量降级为占位或省略，不输出 null。

## 1. 背景

当前平台已经具备日报、通知中心、邮件发送、日报汇总导出、工作流审批、CMDB、Prometheus 告警等能力。现有日报提醒实现偏单点：通过后台定时任务读取 `notify.reminder.*` 配置，在固定时间提醒未提交日报的用户。

客户近期提出的需求集中在周期性运维事项：

1. 每季度季报提醒。
2. 每季度用户核查、密码更新提醒，包括联通云租户新增删除、数据库密码定期更新等。
3. 每季度季度考核，向创建者发送季度考核模板。
4. 每季度监控数据持久化分析输出提醒。
5. 半年报数据归集。
6. 节日后营前巡检提醒。
7. 排班提醒，提醒节假日期间排班负责人及联系方式。
8. 巡检提醒，提醒下周由谁负责发送巡检消息和邮件。

这些需求的共同点不是“日报”，而是“按时间触发的运维计划”。因此需要从产品层面建设“运维日历”，从技术层面建设通用 schedule 能力。

## 2. 产品定位

运维日历用于统一管理周期性运维事项、排班值守、巡检提醒、报表归集和合规核查，形成从计划生成到执行归档的完整闭环。

它面向用户呈现为“日历 + 看板 + 任务详情”，面向系统实现为“计划任务定义 + 实例生成 + 调度触发 + 通知执行 + 留痕复盘”。

### 2.1 不是单纯提醒

运维日历不是只在某个时间发一条通知，而是要回答以下问题：

| 问题 | 运维日历需要提供的能力 |
|---|---|
| 什么时候该做什么 | 日历视图、周视图、今日待办、本周待办 |
| 谁负责 | 负责人、协同人、排班负责人、升级人 |
| 是否收到 | 负责人确认、未确认升级 |
| 是否完成 | 状态流转、执行打卡、完成记录 |
| 做得怎么样 | 执行结果、附件、检查项、异常说明 |
| 后续怎么复盘 | 执行记录、逾期统计、报表素材归集 |

### 2.2 与现有模块关系

| 现有模块 | 结合方式 |
|---|---|
| 通知中心 | 运维日历统一调用通知中心发送站内信和邮件 |
| 日报系统 | 日报提醒迁移为运维日历内置任务；日报数据可作为季报、半年报素材 |
| 报表导出 | 季报、半年报归集可复用日报汇总能力 |
| CMDB | 巡检、密码更新、监控分析任务可关联 CI 实例 |
| Prometheus 告警 | 监控数据分析和巡检可关联告警数据 |
| 变更文档 | 巡检或保障任务发现问题后可关联变更文档 |
| 用户和组织 | 负责人、接收人、组长升级、排班人员来源 |

## 3. 目标与非目标

### 3.1 产品目标

1. 统一承载周期性运维提醒，减少散落在配置项、人工表格、口头通知里的事项。
2. 用日历方式呈现运维节奏，让用户直观看到今日、本周、本月、季度的任务。
3. 建立执行闭环，让提醒变成可确认、可完成、可留痕、可追责的任务。
4. 支撑季度、半年、节假日、周巡检、排班等高频客户场景。
5. 沉淀运维数据，为季报、半年报、考核、审计和复盘提供素材。

### 3.2 研发目标

1. 将现有硬编码日报提醒演进为通用 schedule 机制。
2. 支持多种触发规则：一次性、每日、每周、每月、每季度、每半年、每年、cron、自定义节假日相对日期。
3. 支持任务实例化和幂等执行，避免重复通知和重复生成任务。
4. 支持可扩展任务处理器，后续新增业务类型不需要改动调度核心。

### 3.3 非目标

1. 第一版不实现完整项目管理系统，不做复杂甘特图。
2. 第一版不做强工作流编排，状态流转以轻量闭环为主。
3. 第一版不自动生成复杂经营分析报告，只提供报表素材归集和提醒入口。
4. 第一版不强依赖外部节假日 API，可先支持手工维护节假日历，后续再扩展导入。

## 5. 核心业务闭环

### 5.1 闭环链路

```text
计划定义
  -> 自动生成日历任务
  -> 提前提醒
  -> 负责人确认
  -> 执行检查项
  -> 填写结果和附件
  -> 异常升级
  -> 完成归档
  -> 周/月/季统计
```

### 5.2 状态设计

| 状态 | 说明 | 触发方式 |
|---|---|---|
| 待生成 | 周期规则已创建，但未来实例尚未生成 | 系统内部状态 |
| 待确认 | 任务实例已生成，等待负责人确认收到 | 系统生成 |
| 未开始 | 已确认，但未到计划开始时间 | 用户确认 |
| 进行中 | 到达开始时间或负责人手动开始 | 系统或手动 |
| 已完成 | 负责人提交执行结果 | 手动 |
| 已逾期 | 超过截止时间仍未完成 | 系统 |
| 异常关闭 | 任务因异常无法完成，记录原因 | 手动 |
| 已取消 | 任务取消，不纳入逾期 | 管理员或创建者 |

### 5.3 状态流转

```text
待确认 -> 未开始 -> 进行中 -> 已完成
待确认 -> 已逾期
未开始 -> 已逾期
进行中 -> 已逾期
待确认/未开始/进行中 -> 异常关闭
待确认/未开始/进行中 -> 已取消
```

## 7. 功能需求

## 7.1 运维日历视图

### 7.1.1 月历视图

月历视图用于展示一个自然月内所有计划事项。

必备能力：

1. 支持按月份切换。
2. 每个日期格展示当天任务摘要。
3. 任务按类型显示颜色和图标。
4. 支持筛选任务类型、状态、负责人、组织。
5. 点击日期打开当日工作项放大视图。
6. 点击任务打开任务详情。
7. 对逾期任务给出明显状态标识。
8. 当前日期突出显示。

任务类型建议：

| 类型 | 示例 | 视觉建议 |
|---|---|---|
| 巡检 | 周巡检、节前巡检、节后巡检 | 蓝色 / CheckCircle |
| 排班 | 节假日值班、夜间值守 | 青色 / CalendarClock |
| 报表 | 季报、半年报、日报汇总 | 紫色 / FileText |
| 合规 | 用户核查、密码更新 | 橙色 / ShieldCheck |
| 监控 | 监控数据归档、告警分析 | 红色 / Activity |
| 日报 | 日报提交提醒 | 绿色 / ClipboardList |
| 其他 | 临时提醒 | 灰色 / Bell |

### 7.1.2 视图范围与个性化展示

月历视图本身是同一个页面组件，但不同用户登录后默认看到的数据范围应不同。系统通过权限、组织关系、任务参与关系和视图范围参数控制展示内容。

默认视图建议：

| 用户类型 | 默认视图 | 展示内容 |
|---|---|---|
| 普通运维人员 | 我的日历 | 我负责、我协同、我接收提醒、我创建的任务 |
| 巡检负责人 | 我的日历 | 我的巡检、排班、待确认、待完成任务优先展示 |
| 组长 | 本组日历 | 本组所有任务，并突出逾期、未确认、今日到期任务 |
| 管理员 | 全局日历 | 全平台所有运维任务，可切换到我的或本组 |
| 审计/管理人员 | 全局日历或统计视图 | 关注季度、半年、节假日保障和执行留痕 |

支持的视图范围：

| 视图范围 | 可见人群 | 说明 |
|---|---|---|
| 我的日历 | 所有用户 | 只看与当前用户相关的任务 |
| 本组日历 | 组长、管理员、具备 `ops_calendar:read_group` 的用户 | 查看本组任务 |
| 全局日历 | 管理员、具备 `ops_calendar:read_all` 的用户 | 查看全平台任务 |
| 排班日历 | 相关人员、组长、管理员 | 查看值班安排、节假日负责人和联系方式 |
| 公共运维日历 | 所有具备读取权限的用户 | 展示重大节假日保障、统一季报截止、统一密码更新窗口等公共节点 |

任务可见性建议：

1. 与当前用户有关的任务正常显示。
2. 公共任务即使不是当前用户负责，也可在“我的日历”中以淡色或公共标签展示。
3. 无权限查看详情的公共任务，只展示标题、日期、类型和公共说明，不展示敏感执行结果。
4. 管理员可以在我的、本组、全局、排班、公共视图之间切换。
5. 系统应记住用户上次选择的视图范围，但首次进入按角色默认。

接口建议：

```text
GET /api/ops-calendar/tasks?startDate=2026-07-01&endDate=2026-07-31&scope=mine
GET /api/ops-calendar/tasks?startDate=2026-07-01&endDate=2026-07-31&scope=group
GET /api/ops-calendar/tasks?startDate=2026-07-01&endDate=2026-07-31&scope=all
GET /api/ops-calendar/tasks?startDate=2026-07-01&endDate=2026-07-31&scope=roster
GET /api/ops-calendar/tasks?startDate=2026-07-01&endDate=2026-07-31&scope=public
```

任务模型建议增加：

| 字段 | 说明 |
|---|---|
| visibility | private、group、public |
| public_summary | 公共视图展示摘要 |
| sensitive | 是否包含敏感执行内容 |

这样可以同时满足“每个人看到自己的运维节奏”和“全员知道重要公共节点”的需求。

### 7.1.3 周视图

周视图用于高频执行场景，重点展示本周和下周事项。

必备能力：

1. 展示周一到周日的任务。
2. 支持“本周 / 下周”快捷切换。
3. 任务展示开始时间、负责人、联系方式、状态。
4. 支持快速确认、开始、完成。
5. 下周巡检提醒应优先在周视图可见。

### 7.1.4 列表视图

列表视图用于管理和排查。

字段建议：

| 字段 | 说明 |
|---|---|
| 任务名称 | 实例名称或规则名称 |
| 类型 | 巡检、排班、报表、合规等 |
| 计划时间 | 开始时间和截止时间 |
| 负责人 | 主负责人 |
| 接收人 | 通知接收范围 |
| 状态 | 待确认、进行中、已完成、已逾期等 |
| 来源 | 手动创建、周期规则生成、节假日生成 |
| 最近提醒时间 | 最近一次通知发送时间 |
| 操作 | 查看、确认、完成、取消、重发提醒 |

### 7.1.5 我的任务看板

我的任务看板用于提升执行效率。

卡片建议：

1. 今日待办。
2. 本周待办。
3. 待我确认。
4. 我负责的逾期任务。
5. 我创建的任务。
6. 本组逾期任务，组长和管理员可见。

## 7.2 任务详情

任务详情是闭环的核心页面。

### 7.2.1 基础信息

字段：

| 字段 | 说明 |
|---|---|
| 任务标题 | 展示在日历上的名称 |
| 任务类型 | 巡检、排班、报表、合规、监控等 |
| 状态 | 当前任务状态 |
| 计划开始时间 | 可为空，适用于全天任务 |
| 截止时间 | 用于逾期判断 |
| 负责人 | 主负责人 |
| 协同人 | 可参与执行的人 |
| 接收人 | 通知接收范围 |
| 升级人 | 逾期或未确认时通知的人 |
| 关联对象 | CI、日报、告警、变更文档、文件等 |
| 来源规则 | 如果由周期规则生成，展示规则名称 |

### 7.2.2 确认收到

负责人收到提醒后可以点击“确认收到”。

规则：

1. 只有负责人、协同人、管理员可确认。
2. 确认后记录确认人和确认时间。
3. 已确认任务不重复进入“待确认”统计。
4. 超过确认期限未确认，触发升级提醒。

### 7.2.3 执行清单

每类任务可以绑定 SOP 检查项模板。

检查项字段：

| 字段 | 说明 |
|---|---|
| 检查项名称 | 例如“确认数据库备份状态” |
| 是否必填 | 必填项未完成不能提交 |
| 结果类型 | 勾选、文本、数字、附件、枚举 |
| 默认说明 | 引导执行人员填写 |
| 排序 | 控制展示顺序 |

典型模板：

**巡检模板**

1. 检查核心应用状态。
2. 检查数据库连接和慢查询情况。
3. 检查磁盘、CPU、内存使用率。
4. 检查 Prometheus 当前告警。
5. 检查备份任务最近执行情况。
6. 记录异常和处理建议。

**密码更新模板**

1. 核查联通云租户新增、删除、权限变更。
2. 核查数据库账号清单。
3. 更新数据库密码或确认未到更新周期。
4. 更新应用连接串或记录无需更新原因。
5. 验证业务连接正常。
6. 上传核查记录。

**季报模板**

1. 归集本季度日报汇总。
2. 归集本季度巡检完成情况。
3. 归集本季度告警统计。
4. 归集本季度变更文档。
5. 填写问题复盘和改进计划。

### 7.2.4 执行结果

任务完成时必须填写执行结果。

字段：

| 字段 | 说明 |
|---|---|
| 执行结论 | 正常、异常、部分完成、无需执行 |
| 结果说明 | 文本说明 |
| 风险等级 | 无、低、中、高 |
| 附件 | 截图、Excel、巡检报告等 |
| 关联日报 | 可关联当天或相关日报 |
| 关联 CI | 可关联涉及的配置项 |
| 关联告警 | 可关联 Prometheus 告警 |
| 关联变更 | 可关联后续变更文档 |

### 7.2.5 动态记录

记录任务全生命周期：

1. 创建任务。
2. 发送提醒。
3. 负责人确认。
4. 状态变更。
5. 执行结果提交。
6. 附件上传。
7. 逾期升级。
8. 取消或异常关闭。

## 7.3 周期规则

周期规则用于自动生成日历任务。

### 7.3.1 规则字段

| 字段 | 说明 |
|---|---|
| 规则名称 | 例如“季度密码更新提醒” |
| 任务类型 | 巡检、排班、报表、合规等 |
| 启用状态 | 启用、停用 |
| 触发方式 | 一次性、每日、每周、每月、每季度、每半年、每年、cron、节假日相对日期 |
| 触发时间 | 具体日期、星期、月份、季度位置或 cron |
| 提前生成天数 | 提前多少天生成任务实例 |
| 提前提醒规则 | 提前 N 天、提前 N 小时、到期当天 |
| 截止规则 | 触发后 N 天或指定时间 |
| 负责人规则 | 固定用户、组长、排班表、轮值规则、创建者 |
| 接收人规则 | 负责人、协同人、组成员、管理员、自定义用户 |
| 升级规则 | 未确认或逾期后通知谁 |
| 通知模板 | 标题和正文模板 |
| SOP 模板 | 执行检查项模板 |
| 关联对象规则 | 是否自动关联 CI、告警、日报等 |

### 7.3.2 支持的周期

| 周期 | 示例 |
|---|---|
| 一次性 | 2026-07-01 10:00 触发 |
| 每日 | 每个工作日 17:00 日报提醒 |
| 每周 | 每周五 16:00 提醒下周巡检负责人 |
| 每月 | 每月最后一个工作日生成月度巡检归档 |
| 每季度 | 每季度最后 5 个工作日生成季报提醒 |
| 每半年 | 每半年最后 10 个工作日生成半年报归集 |
| 每年 | 每年固定日期执行年度账号核查 |
| cron | 高级用户自定义 cron |
| 节假日相对 | 节前 2 个工作日、节后第 1 个工作日、营前第 1 天 |

### 7.3.3 规则预览

创建或编辑周期规则时，系统应展示未来 3 到 6 次触发时间预览。

示例：

| 触发批次 | 计划日期 | 任务名称 |
|---|---|---|
| 下一次 | 2026-09-25 | 2026 Q3 季报提醒 |
| 第 2 次 | 2026-12-25 | 2026 Q4 季报提醒 |
| 第 3 次 | 2027-03-25 | 2027 Q1 季报提醒 |

排班管理用于支撑值班提醒、节假日负责人提醒和巡检负责人提醒。

1. 节假日前发送完整值班表。

## 8. 页面设计建议

## 8.1 导航入口

运维日历应作为侧边栏一级菜单展示，不归入流程中心、CMDB、系统管理或其他二级菜单。

```text
运维日历
```

设计理由：

1. 运维日历承载日报提醒、巡检、排班、季报、半年报、合规核查、监控分析等跨模块事项，不属于单一业务模块。
2. 使用频次高，普通运维人员、组长、管理员都会日常进入。
3. 它是“今天/本周该做什么”的入口，应具备和工作台、CMDB、日报同级的导航权重。
4. 作为一级菜单有利于后续扩展排班、节假日历、统计复盘等子能力。

建议侧边栏位置：

```text
工作台
运维日历
CMDB
日报审批
...
```

## 8.2 工作台集成

工作台中建议用“运维日历”卡片替换现有“近期 CMDB 变更”卡片。

### 8.2.1 替换理由

1. 近期 CMDB 变更偏信息展示，用户处理动作较少。
2. 运维日历更接近日常工作入口，能展示今日待办、逾期、待确认、值班和巡检。
3. 工作台需要优先展示“需要我行动”的内容，运维日历卡片更符合高频使用场景。
4. CMDB 变更仍可在 CMDB 模块、实例详情、变更历史中查看，不需要占据首页核心卡片。

### 8.2.2 工作台卡片内容

卡片名称建议：

```text
运维日历
```

卡片内容：

| 区域 | 内容 |
|---|---|
| 顶部摘要 | 今日任务数、待确认数、逾期数 |
| 今日列表 | 展示今日 3 到 5 条最重要任务 |
| 明日/本周提示 | 展示即将到期、节假日值班、下周巡检 |
| 状态标识 | 待确认、进行中、逾期、已完成 |
| 快捷入口 | 查看日历、我的任务、逾期任务 |

排序建议：

1. 已逾期。
2. 今日到期。
3. 待确认。
4. 我负责的进行中任务。
5. 公共重要节点。

卡片操作：

1. 点击卡片标题进入运维日历一级页面。
2. 点击具体任务打开任务详情。
3. 点击日期或“今日”打开当日工作项放大视图。

## 8.3 页面结构

运维日历首页建议布局：

```text
顶部：标题、月份切换、视图切换、筛选、创建按钮

主体：
  左侧/主区域：月历或周视图
  右侧：今日待办、本周逾期、待确认

抽屉：
  点击日期 -> 当日工作项放大视图
  点击任务 -> 任务详情
```

### 8.3.1 顶部操作区

控件：

1. 今天。
2. 上一月 / 下一月。
3. 月视图 / 周视图 / 列表。
4. 类型筛选。
5. 状态筛选。
6. 我的 / 本组 / 全部。
7. 新建任务。
8. 新建周期规则，管理员可见。

### 8.3.2 当日工作项放大视图

点击月历中的某一天后，应展示该日期的放大工作项视图，而不是只弹出简单列表。

展示形式建议：

1. 使用右侧大抽屉或居中弹层，宽度应足够展示任务分组和操作。
2. 顶部展示日期、星期、是否节假日、当日任务统计。
3. 按状态或类型分组展示任务。
4. 支持在放大视图内直接确认、开始、完成、重发提醒。
5. 支持切换前一天、后一天，不关闭视图。
6. 对排班任务展示负责人、备份负责人和联系方式。
7. 对公共任务展示公共摘要，对敏感任务按权限脱敏。

分组建议：

| 分组 | 说明 |
|---|---|
| 待处理 | 待确认、未开始、进行中、今日到期 |
| 已逾期 | 超过截止时间未完成 |
| 排班值守 | 当日值班、节假日负责人 |
| 公共节点 | 季报截止、统一密码更新窗口、节假日保障 |
| 已完成 | 当日已完成任务，默认折叠 |

当日工作项字段：

| 字段 | 说明 |
|---|---|
| 时间 | 开始时间、截止时间或全天 |
| 标题 | 任务名称 |
| 类型 | 巡检、排班、报表、合规等 |
| 负责人 | 主负责人 |
| 联系方式 | 排班和值守类任务优先展示 |
| 状态 | 当前状态 |
| 操作 | 确认、开始、完成、查看详情 |

### 8.3.3 创建任务入口

支持两类创建：

1. 新建临时任务。
2. 新建周期规则。

创建临时任务用于一次性提醒；创建周期规则用于季度、半年、周巡检、节假日等重复任务。

## 8.4 任务详情抽屉

建议用右侧抽屉，避免离开日历上下文。

区块：

1. 标题和状态。
2. 基础信息。
3. 负责人和接收人。
4. 执行清单。
5. 执行结果。
6. 附件和关联对象。
7. 动态记录。

操作按钮：

| 状态 | 操作 |
|---|---|
| 待确认 | 确认收到、取消 |
| 未开始 | 开始执行、取消 |
| 进行中 | 提交完成、异常关闭 |
| 已逾期 | 提交完成、申请延期、异常关闭 |
| 已完成 | 查看结果、导出记录 |

## 12. 调度设计建议

## 12.1 核心设计

建议新增统一调度器：

```text
OpsCalendarScheduler
```

职责：

1. 定期扫描启用中的周期规则。
2. 为未来时间窗口生成任务实例。
3. 扫描需要提醒的任务。
4. 扫描未确认、逾期任务并触发升级。
5. 记录每次执行日志。

## 12.2 Handler 扩展

不同业务动作通过 handler 扩展：

| Handler | 说明 |
|---|---|
| DailyReportReminderHandler | 日报提交提醒 |
| QuarterReportReminderHandler | 季报提醒 |
| PasswordRotationReminderHandler | 密码更新提醒 |
| QuarterAssessmentHandler | 季度考核模板发送 |
| MonitoringAnalysisReminderHandler | 监控数据分析提醒 |
| SemiAnnualAggregationHandler | 半年报归集 |
| HolidayInspectionHandler | 节前、节后、营前巡检 |
| DutyRosterReminderHandler | 排班提醒 |
| WeeklyInspectionReminderHandler | 下周巡检负责人提醒 |

调度核心只处理“什么时候触发”，handler 处理“触发后做什么”。

## 12.3 幂等要求

必须避免重复生成和重复通知：

1. 同一规则、同一计划时间只能生成一个任务实例。
2. 同一任务、同一提醒阶段只能发送一次通知。
3. 调度器异常重启后可以继续扫描，不依赖内存状态。
4. 所有执行结果写入任务日志。
5. 调度扫描按 `next_generate_at <= now` 过滤规则，不做全表扫描（见技术规格 6.2，配合 `idx_ops_rule_next_generate` 索引）。
6. **平台审计**：所有写操作（建/改/删规则、排班、节假日、任务状态流转、取消、异常关闭）除写任务日志外，必须在同一事务内写平台 `audit_log`（项目强制规则 2）。两者分工：任务日志=领域时间线，audit_log=合规审计。调度器系统操作 `operatorId=0`。

## 15. 验收标准

### 15.1 日历展示

1. 用户可以在月历中看到当月任务。
2. 用户可以在周视图中看到本周和下周任务。
3. 用户可以按类型、状态、负责人筛选任务。
4. 点击任务可以查看详情和动态记录。

### 15.2 任务闭环

1. 负责人可以确认收到任务。
2. 负责人可以开始任务并提交完成。
3. 必填检查项未完成时不能提交。
4. 任务逾期后状态自动变为已逾期。
5. 逾期任务会触发升级提醒。
6. 任务完成后保留执行结果、附件和日志。

### 15.3 周期规则

1. 管理员可以创建季度季报提醒规则。
2. 管理员可以创建每周巡检提醒规则。
3. 管理员可以创建半年报归集规则。
4. 规则可以预览未来触发时间。
5. 系统能够按规则生成任务实例。
6. 禁用规则后不再生成新任务。

### 15.4 排班和节假日

1. 管理员可以维护节假日。
2. 管理员可以维护节假日排班。
3. 系统可以提醒节假日期间负责人和联系方式。
4. 系统可以提示无人值班或缺少联系方式。

### 15.5 报表归集

1. 用户可以查看季度任务完成统计。
2. 用户可以查看半年报素材清单。
3. 用户可以跳转到关联日报、巡检任务、告警和变更文档。

## 17. 风险与注意事项

| 风险 | 说明 | 建议 |
|---|---|---|
| 任务重复生成 | 调度器重启或多实例部署可能重复生成 | 使用唯一约束和幂等 key |
| 通知重复发送 | 扫描式调度容易重复提醒 | 记录提醒阶段和发送记录 |
| cron 表达式复杂 | 当前日报提醒只支持简单 cron | 使用 Spring CronExpression 或成熟解析库 |
| 节假日规则复杂 | 法定节假日、调休、公司特殊日期差异大 | 第一版手工维护，后续支持导入 |
| 权限边界 | 本人、本组、全部视图不同 | 明确 read/read_group/read_all |
| 闭环负担过重 | 所有任务都强制填很多字段会影响使用 | 按任务类型配置必填项 |
| 日历信息过载 | 一天任务太多会拥挤 | 日期格只显示摘要，详情放抽屉和列表 |
| 调度器全表扫描浪费 | 每分钟遍历所有规则成本高 | 已决议（2026-06-30）：按 `next_generate_at <= now` 过滤，命中 `idx_ops_rule_next_generate` 索引；详见 spec 6.2 |
| 写操作未留痕 | 平台强制规则要求所有写操作写 `audit_log` | 已决议（2026-06-30）：规则/任务/排班/节假日所有写操作同事务写 `audit_log`，与 `ops_schedule_task_log` 并存；详见 spec 7.4 |
| 日历被日报任务淹没 | 每人每天一条可闭环任务会污染日历 | 已决议（2026-06-30）：每天一条汇总跟踪任务 + 向未提交者扇出通知，日报不降级；详见 spec 9.2 |
