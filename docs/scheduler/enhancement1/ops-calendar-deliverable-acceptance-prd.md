# 运维日历增强 1 PRD：任务附件与交付物验收

> 模块：运维日历增强能力
> 状态：产品设计稿
> 日期：2026-06-30
> 定位：在已完成的运维日历基础上，增强“任务派发附件、执行交付物、验收确认、驳回整改”的闭环能力。

---

## 1. 背景

当前运维日历已经完成周期性任务的计划、提醒、执行、留痕和统计闭环。客户后续提出一个更细的管理诉求：

1. 管理员或创建者派发周期任务时，可能需要附带模板、SOP、表格、参考材料。
2. 负责人完成任务时，可能不能只填写文字说明，而是必须上传一个或多个交付附件。
3. 某些任务需要经过验收人认可后才算真正完成。
4. 如果验收不通过，验收人需要驳回并填写理由，负责人按要求修改后重新提交。

该需求不是重新建设工单系统，而是在运维日历的任务闭环上增加“交付物验收”能力。

---

## 2. 产品定位

本增强用于解决以下问题：

| 问题 | 增强能力 |
|---|---|
| 派发任务时需要给负责人模板或说明 | 支持派发附件、模板附件、参考材料 |
| 负责人需要提交证据 | 支持交付附件、执行证据、关联材料 |
| 管理人员需要确认质量 | 支持验收人、验收策略、验收意见 |
| 驳回后需要明确整改要求 | 驳回原因必填，支持修改建议和重新提交截止时间 |
| 多轮提交需要留痕 | 支持交付版本和验收日志 |

一句话目标：

```text
负责人提交证据，验收人确认质量，系统保留全过程证据链。
```

---

## 3. 范围

### 3.1 本次增强范围

1. 任务派发时上传附件。
2. 周期规则配置模板附件或参考附件。
3. 负责人提交任务交付物。
4. 支持“提交验收、验收通过、验收驳回、重新提交”。
5. 驳回时填写原因和修改建议。
6. 支持多轮提交版本记录。
7. 支持待我验收列表或筛选。
8. 支持验收提醒和验收逾期提醒。
9. 支持附件权限和敏感附件控制。

### 3.2 非目标

1. 不建设独立工单系统。
2. 不做通用客服工单、故障工单、ITSM 派单池。
3. 不做复杂多级审批流。
4. 不替代现有流程中心。
5. 不要求所有运维日历任务都必须验收。

---

## 4. 角色

| 角色 | 诉求 |
|---|---|
| 管理员 | 创建需要交付物和验收的周期规则，配置模板附件和验收人 |
| 创建者 | 派发临时任务，上传派发附件，查看验收结果 |
| 负责人 | 下载模板、执行任务、上传交付物、重新提交 |
| 协同人 | 协助上传材料或补充执行说明 |
| 验收人 | 查看交付物，填写通过或驳回意见 |
| 组长 | 关注本组待验收、被驳回、逾期未提交任务 |
| 审计/管理人员 | 查看提交版本、验收记录、驳回原因和附件留痕 |

---

## 5. 核心业务流程

### 5.1 需要验收的任务流程

```text
任务生成或派发
  -> 负责人确认
  -> 执行任务
  -> 上传交付物
  -> 提交验收
  -> 验收人审核
  -> 验收通过
  -> 任务完成归档
```

### 5.2 驳回整改流程

```text
提交验收
  -> 验收驳回
  -> 填写驳回原因和修改建议
  -> 通知负责人
  -> 负责人修改交付物
  -> 重新提交验收
  -> 验收通过或再次驳回
```

## 6. 状态设计

### 6.1 任务主状态

任务主状态可以沿用运维日历已有状态，不必重新设计完整工单状态。

建议在原任务状态基础上增加一个独立字段：

```text
acceptance_status
```

### 6.2 验收状态

| 状态 | 说明 |
|---|---|
| not_required | 不需要验收 |
| pending_submit | 等待负责人提交交付物 |
| pending_acceptance | 已提交，等待验收 |
| accepted | 验收通过 |
| rejected | 验收驳回，等待修改 |

### 6.3 状态关系

```text
任务进行中 + pending_submit
  -> 提交验收
任务进行中 + pending_acceptance
  -> 验收通过
任务已完成 + accepted

任务进行中 + pending_acceptance
  -> 验收驳回
任务进行中 + rejected
  -> 重新提交验收
任务进行中 + pending_acceptance
```

规则：

1. 不需要验收的任务，负责人提交执行结果后可直接完成。
2. 需要验收的任务，负责人提交后进入“待验收”，不能直接变为已完成。
3. 验收通过后，任务才进入“已完成”。
4. 验收驳回后，任务回到“进行中”或保持“进行中 + rejected”。
5. 驳回任务重新提交后，生成新的交付版本。

## 7. 附件分类

附件不应混在一个列表里展示，需要按用途区分。

| 附件类型 | 上传人 | 用途 |
|---|---|---|
| 派发附件 | 管理员、创建者 | 任务要求、说明文件、一次性参考材料 |
| 模板附件 | 管理员、创建者 | 季报模板、巡检模板、考核模板、Excel 表格 |
| 参考附件 | 管理员、创建者 | 历史报告、SOP、截图示例、制度文件 |
| 交付附件 | 负责人、协同人 | 执行结果、报告、截图、清单 |
| 证据附件 | 负责人、协同人 | 操作截图、验证截图、导出数据 |
| 验收附件 | 验收人 | 批注文件、审核版文档、补充说明 |

建议字段：

```text
attachment_role:
instruction
template
reference
deliverable
evidence
acceptance
```

## 8. 功能需求

## 8.1 周期规则附件

管理员创建或编辑周期规则时，可以上传模板附件和参考附件。

典型场景：

| 规则 | 附件 |
|---|---|
| 季报提醒 | 季报模板.docx、季度指标口径.xlsx |
| 半年报归集 | 半年报模板.docx、素材清单.xlsx |
| 季度密码更新 | 密码更新记录模板.xlsx |
| 监控数据分析 | 监控分析报告模板.docx |
| 巡检任务 | 巡检报告模板.xlsx、巡检 SOP.pdf |

规则：

1. 周期规则附件不直接复制文件内容。
2. 任务生成时引用规则附件。
3. 如果规则附件后续更新，新生成任务使用最新附件。
4. 已生成任务默认保留当时引用的附件版本或文件 ID。
5. 是否需要冻结附件版本由文件模块能力决定，MVP 可先保留文件 ID 引用。

## 8.2 临时任务派发附件

创建临时任务时，创建者可以上传派发附件。

规则：

1. 派发附件属于当前任务。
2. 负责人和协同人可查看和下载。
3. 无详情权限的公共视图不能下载附件。
4. 敏感任务附件必须跟随任务详情权限控制。

## 8.3 交付物要求配置

规则或任务可配置交付物要求。

字段建议：

| 字段 | 说明 |
|---|---|
| deliverable_required | 是否必须提交交付物 |
| min_deliverable_count | 最少附件数量 |
| allowed_file_types | 允许的文件类型 |
| max_file_size_mb | 单文件大小限制 |
| result_summary_required | 是否必须填写结果说明 |
| linked_object_required | 是否必须关联日报、CI、告警或变更 |
| deliverable_description | 交付要求说明 |

示例：

```text
必须上传至少 1 个 PDF 或 Word 文件，并填写不少于 20 字的分析结论。
```

## 8.4 提交验收

负责人或协同人完成执行后，点击“提交验收”。

提交内容：

1. 执行说明。
2. 交付附件。
3. 证据附件。
4. 关联日报、CI、告警、变更文档。
5. 风险等级。
6. 补充备注。

校验：

1. 如果 `deliverable_required=true`，必须满足最少附件数量。
2. 如果限制文件类型，只允许上传指定类型。
3. 如果要求结果说明，结果说明不能为空。
4. 如果要求关联对象，必须至少关联一个指定对象。

提交后：

1. `acceptance_status` 改为 `pending_acceptance`。
2. 生成一条交付版本记录。
3. 通知验收人。
4. 写任务动态和审计日志。

## 8.5 验收通过

验收人查看交付物后，可点击“验收通过”。

通过时可填写：

1. 验收意见。
2. 验收附件，可选。

通过后：

1. `acceptance_status` 改为 `accepted`。
2. 任务状态改为 `completed`。
3. 写入完成时间和完成确认人。
4. 通知负责人和创建者。
5. 写任务动态和审计日志。

## 8.6 验收驳回

验收人驳回时，必须填写驳回原因。

驳回字段：

| 字段 | 必填 | 说明 |
|---|---:|---|
| reject_reason | 是 | 驳回原因 |
| revision_suggestion | 否 | 修改建议 |
| required_resubmit | 否 | 是否要求重新提交 |
| resubmit_due_at | 否 | 重新提交截止时间 |
| acceptance_attachments | 否 | 批注附件或审核材料 |

驳回后：

1. `acceptance_status` 改为 `rejected`。
2. 任务保持进行中。
3. 通知负责人和协同人。
4. 驳回原因展示在任务详情醒目位置。
5. 写任务动态和审计日志。

规则：

1. 驳回原因不能为空。
2. 批量驳回不建议支持，避免无明确理由。
3. 如果设置重新提交截止时间，超过后进入逾期统计或整改逾期统计。

## 8.7 多轮提交版本

每次提交验收都生成一个交付版本。

版本信息：

| 字段 | 说明 |
|---|---|
| version_no | 第几次提交 |
| submitter_id | 提交人 |
| submit_summary | 提交说明 |
| risk_level | 风险等级 |
| submitted_at | 提交时间 |
| status | pending、accepted、rejected |

版本规则：

1. 第一次提交为 V1。
2. 驳回后重新提交为 V2。
3. 每个版本保留当时提交的附件和关联对象。
4. 验收记录关联到具体版本。

## 8.8 待我验收

新增筛选或看板：

```text
待我验收
```

展示内容：

1. 任务标题。
2. 任务类型。
3. 提交人。
4. 提交时间。
5. 验收截止时间。
6. 当前版本。
7. 操作：查看、通过、驳回。

建议入口：

1. 运维日历顶部筛选。
2. 我的任务看板。
3. 工作台运维日历卡片。

## 8.9 验收提醒

新增通知时机：

| 时机 | 接收人 |
|---|---|
| 交付物提交 | 验收人 |
| 验收即将逾期 | 验收人 |
| 验收逾期 | 验收人、组长 |
| 验收通过 | 负责人、创建者 |
| 验收驳回 | 负责人、协同人 |
| 重新提交逾期 | 负责人、组长 |

---

## 9. 验收策略

支持以下策略：

| 策略 | 说明 | MVP 建议 |
|---|---|---|
| single | 单个验收人通过即可 | 必做 |
| any | 多个验收人任一通过即可 | 可做 |
| all | 所有验收人都通过才算完成 | 后续 |

MVP 建议：

1. 先支持 `single`。
2. 可预留 `acceptance_policy` 字段。
3. 多人验收先用“任一通过”作为轻量扩展。

## 10. 权限设计

| 操作 | 允许人 |
|---|---|
| 上传规则模板附件 | 管理员、具备规则管理权限的人 |
| 上传派发附件 | 创建者、管理员 |
| 查看派发附件 | 负责人、协同人、验收人、创建者、组长、管理员 |
| 上传交付附件 | 负责人、协同人 |
| 提交验收 | 负责人、协同人 |
| 验收通过 | 验收人、管理员 |
| 验收驳回 | 验收人、管理员 |
| 查看验收记录 | 有任务详情权限的人 |
| 下载敏感附件 | 有敏感任务详情权限的人 |

注意：

1. 公共任务只展示公共摘要，不暴露附件下载。
2. 敏感任务附件必须跟随任务敏感权限。
3. 验收人与负责人相同可以允许，但需要提示风险。
4. 合规类任务建议要求验收人与负责人不同。

## 11. 页面设计建议

## 11.1 任务详情分区

任务详情建议分为三个主要区块：

```text
任务要求
  - 任务说明
  - 模板附件
  - 参考材料

执行提交
  - 执行说明
  - 交付附件
  - 证据附件
  - 关联对象

验收记录
  - 当前验收状态
  - 提交版本
  - 通过/驳回记录
  - 驳回原因和修改建议
```

## 11.2 提交验收弹窗

字段：

1. 执行说明。
2. 风险等级。
3. 上传交付附件。
4. 上传证据附件。
5. 关联日报、CI、告警、变更。
6. 提交按钮。

## 11.3 验收弹窗

操作：

1. 验收通过。
2. 验收驳回。

驳回时展示：

1. 驳回原因，必填。
2. 修改建议。
3. 重新提交截止时间。
4. 上传批注附件。

## 11.4 待我验收列表

字段：

| 字段 | 说明 |
|---|---|
| 任务标题 | 点击进入详情 |
| 类型 | 任务类型 |
| 提交人 | 最近一次提交人 |
| 提交时间 | 最近一次提交时间 |
| 验收截止 | 如果有 |
| 当前版本 | V1、V2 |
| 状态 | 待验收、已驳回、已通过 |
| 操作 | 通过、驳回、查看 |

## 12. 数据模型建议

### 12.1 任务表扩展

建议在任务实例表增加：

| 字段 | 说明 |
|---|---|
| acceptance_required | 是否需要验收 |
| acceptance_status | not_required、pending_submit、pending_acceptance、accepted、rejected |
| acceptance_policy | single、any、all |
| acceptance_due_at | 验收截止时间 |
| accepted_at | 验收通过时间 |
| accepted_by | 最终通过人 |
| last_rejected_at | 最近驳回时间 |
| last_rejected_by | 最近驳回人 |
| last_reject_reason | 最近驳回原因 |

### 12.2 规则表扩展

建议在周期规则表增加：

| 字段 | 说明 |
|---|---|
| acceptance_config | 验收配置 JSON |
| deliverable_config | 交付物要求 JSON |

`acceptance_config` 示例：

```json
{
  "required": true,
  "policy": "single",
  "acceptorRule": {
    "type": "group_leader",
    "groupId": 2
  },
  "acceptanceDueHours": 48
}
```

`deliverable_config` 示例：

```json
{
  "required": true,
  "minCount": 1,
  "allowedFileTypes": ["pdf", "docx", "xlsx", "png"],
  "resultSummaryRequired": true,
  "linkedObjectRequired": false,
  "description": "请上传完整报告和关键截图"
}
```

### 12.3 附件/关联表扩展

可复用任务关联对象表，但需要增加角色字段：

| 字段 | 说明 |
|---|---|
| link_type | file、daily_report、ci_instance、prometheus_alert、change_doc、external |
| link_role | instruction、template、reference、deliverable、evidence、acceptance |
| version_id | 如果属于某次提交版本，关联版本 ID |

### 12.4 交付版本表

建议新增：

```text
ops_schedule_deliverable_version
```

字段：

| 字段 | 说明 |
|---|---|
| id | 主键 |
| task_id | 任务 ID |
| version_no | 版本号 |
| submitter_id | 提交人 |
| submit_summary | 提交说明 |
| risk_level | 风险等级 |
| status | pending、accepted、rejected |
| submitted_at | 提交时间 |

### 12.5 验收记录表

建议新增：

```text
ops_schedule_acceptance_log
```

字段：

| 字段 | 说明 |
|---|---|
| id | 主键 |
| task_id | 任务 ID |
| version_id | 交付版本 ID |
| action | submit、approve、reject |
| operator_id | 操作人 |
| comment | 说明、通过意见或驳回原因 |
| revision_suggestion | 修改建议 |
| required_resubmit | 是否要求重新提交 |
| resubmit_due_at | 重新提交截止时间 |
| created_at | 操作时间 |

规则：

1. `action=reject` 时 `comment` 必填。
2. `action=approve` 时 `comment` 可选。
3. `action=submit` 时记录提交说明。

## 13. API 建议

### 13.1 附件

| 方法 | 路径 | 说明 |
|---|---|---|
| POST | `/api/ops-calendar/tasks/{id}/attachments` | 上传任务附件 |
| GET | `/api/ops-calendar/tasks/{id}/attachments` | 查询任务附件 |
| DELETE | `/api/ops-calendar/tasks/{id}/attachments/{linkId}` | 删除任务附件 |
| POST | `/api/ops-calendar/rules/{id}/attachments` | 上传规则模板或参考附件 |
| GET | `/api/ops-calendar/rules/{id}/attachments` | 查询规则附件 |

### 13.2 交付与验收

| 方法 | 路径 | 说明 |
|---|---|---|
| POST | `/api/ops-calendar/tasks/{id}/submit-acceptance` | 提交验收 |
| POST | `/api/ops-calendar/tasks/{id}/acceptance/approve` | 验收通过 |
| POST | `/api/ops-calendar/tasks/{id}/acceptance/reject` | 验收驳回 |
| GET | `/api/ops-calendar/tasks/{id}/deliverable-versions` | 查询交付版本 |
| GET | `/api/ops-calendar/tasks/{id}/acceptance-logs` | 查询验收记录 |
| GET | `/api/ops-calendar/tasks/pending-acceptance` | 待我验收 |

### 13.3 请求示例

提交验收：

```json
{
  "submitSummary": "已完成季度监控分析，详见附件。",
  "riskLevel": "low",
  "deliverableFileIds": [101, 102],
  "evidenceFileIds": [103],
  "linkedDailyReportIds": [1, 2],
  "linkedCiInstanceIds": [20],
  "linkedAlertIds": [55]
}
```

验收驳回：

```json
{
  "rejectReason": "报告缺少近三个月 CPU 和内存趋势截图。",
  "revisionSuggestion": "请补充趋势截图，并说明高峰期资源使用情况。",
  "requiredResubmit": true,
  "resubmitDueAt": "2026-07-03T18:00:00",
  "acceptanceFileIds": [201]
}
```

验收通过：


## 17. 验收标准

1. 管理员创建规则时可以上传模板附件。
2. 周期任务生成后，负责人能看到规则模板附件。
3. 创建临时任务时可以上传派发附件。
4. 需要验收的任务提交后不会直接完成，而是进入待验收。
5. 验收人可以通过任务，通过后任务完成。
6. 验收人可以驳回任务，驳回原因必填。
7. 被驳回任务负责人可以重新提交。
8. 系统保留每次提交版本和每次验收记录。
9. 公共视图和无详情权限用户不能下载敏感附件。
10. 待我验收列表能展示当前用户需要处理的验收任务。

## 18. 风险与注意事项

| 风险 | 说明 | 建议 |
|---|---|---|
| 功能膨胀为工单系统 | 容易引入派单池、SLA、客户入口 | 本增强只做周期任务交付物验收 |
| 附件权限泄漏 | 模板或交付物可能包含敏感信息 | 附件下载必须走任务详情权限 |
| 只保留最终附件 | 会丢失多轮整改过程 | 必须保留交付版本 |
| 驳回无原因 | 负责人无法整改 | 驳回原因必填 |
| 验收人长期不处理 | 任务卡在待验收 | 增加验收提醒和验收逾期升级 |
| 规则附件复制过多 | 周期任务反复复制模板浪费存储 | 规则附件以引用为主 |
| 负责人和验收人为同一人 | 合规价值降低 | 允许但提示风险，合规任务建议限制 |

## 19. 待确认问题

1. 文件上传是否直接复用现有共享文件能力，还是需要任务私有附件能力。
2. 规则模板附件是否需要文件版本冻结。
3. 是否允许负责人删除已提交版本中的附件。
4. 验收通过后是否允许补充附件。
5. 是否需要在线预览 Word、PDF、Excel。
6. 合规类任务是否强制验收人与负责人不能相同。
7. 验收超时是否计入任务逾期率，还是单独统计为验收逾期。
